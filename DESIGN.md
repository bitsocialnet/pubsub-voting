# Design: @bitsocial/pubsub-votes

This document is the design of record. The engine and client lifecycle are implemented — schemas, canonical encoding, topic/manifest derivation, the verify pipeline, LWW resolution, the tally, the transport's **validate-before-forward gossip gate** (`src/transport/gossip-validator.ts`), the `PubsubVoter` client-level republish scheduler (`src/client/voter.ts`), and durable vote-intent persistence (Node SQLite / browser IndexedDB, `src/store/`). What remains is host-blocked deferred work: cold-start winner-CID sync over the libp2p fetch protocol and checkpoint compaction. This document explains what each part is for and why.

**State-model migration in progress.** This document describes the *target* state model — a grow-only, union-only **LWW winner-set** keyed by wallet, synced by gossiping individual bundle CIDs and reconciled by **checkpoints** (see "CRDT" and "Checkpoints"). The shipped code still carries the earlier **parent-linked Merkle-CRDT** (each bundle links the heads it saw; peers gossip head CIDs and walk parents). Removing the parent links (`crdt/`, `crdt/codec.ts`, `transport/gossip-validator.ts`, and their tests) is a pending follow-up; because v1 is not deployed, the resulting bundle-CID change carries no migration cost.

## Context

- **Consumer:** [5chan](https://github.com/bitsocialnet/5chan) and its [competitive directory system](https://github.com/bitsocialnet/5chan/blob/master/README.md#competitive-directory-system). Many boards compete for each directory slot; only the highest-voted board appears on the homepage. Current curation is manual pull requests to [`5chan-directories.json`](https://github.com/bitsocialnet/lists/blob/master/5chan-directories.json). This library replaces that with holder-decided, censorship-resistant voting.
- **Origin:** [pkc-js issue #25](https://github.com/pkcprotocol/pkc-js/issues/25), which sketched pubsub voting (settings in the topic, bucketized block numbers, per-bucket heartbeat republish). This design keeps the good parts and replaces the bare heartbeat with a CRDT — a grow-only LWW winner-set reconciled by checkpoints.
- **Host:** [pkc-js](https://github.com/pkcprotocol/pkc-js) (Public Key Communities) provides the shared libp2p/Helia node. It is *only* the node host: it supplies no identity and no signing for voting. This library adds everything chain-related, the CRDT, and the vote-signing/verification, none of which exist in pkc-js.

## Decisions

- **State model: grow-only LWW winner-set + checkpoints.** Votes form a last-write-wins element-set keyed by wallet (highest `blockNumber`, tiebreak lowest bundle CID). The live path gossips **individual bundle CIDs**; each peer verifies a bundle and LWW-merges it into its winner-set. Cold-start and compaction use **checkpoints** — a canonical flat array of the current winners, fetched from several peers and reconciled by union + LWW. Precedent for leaderless CRDT convergence over pubsub: OrbitDB, Secure Scuttlebutt, and Merkle-CRDTs / `go-ds-crdt` (Sanjuán et al., Protocol Labs 2020; powers IPFS Cluster's shared pinset with no leader) — though those sync via a Merkle-clock DAG where this design uses checkpoint reconciliation instead.
- **Weighting v1: 5chan Pass only.** ERC-721 ownership is the gate; weight is constant (1 pass = 1 vote). The rule design keeps a combo path open (Pass gate + BSO ERC-20 weight) without a redesign.
- **Voting is upvote-only in v1.** Downvotes turn a contest into a cheap weapon against competitors; approval-style upvotes (vote for as many boards as you like, capped by `maxVotesPerAddress`) avoid that. The wire keeps a numeric `vote` field so a future criteria could widen the range.
- **Identity is the voting wallet, not a pkc-js author.** The gating-chain wallet (whose private key 5chan or its clients hold, never pkc-js) signs each vote directly as EIP-712 typed data; the address recovered from that signature is the only identity in the system. There is no pkc-js author and no author→wallet binding — the signature *is* the identity. This drops a whole verification layer (no binding to forge or keep monotonic) and the ed25519 author signer.

## Votes are off-chain

A vote is never written to a blockchain. Casting a vote is: sign the bundle with the gating-chain wallet (EIP-712 typed data — no gas, an off-chain signature), store it as a dag-cbor bundle in the host's Helia blockstore (IPFS content-addressing, not a chain), and broadcast the new bundle's CID over gossipsub. There is no transaction and no gas.

The chain is **read-only**, and only to decide *who may vote* and *how much*:

- **Gate** (the `rule` slot, `erc721-min-balance`): does the wallet hold the 5chan Pass? This is the Sybil-resistance anchor. Without a scarce, hard-to-forge asset gating votes, anyone could generate unlimited identities and stuff the ballot.
- **Weight** (`constant` in v1, which reads no chain state at all; `erc20-balance` reserved for the combo path).

`ChainClient` is just a viem `PublicClient` — read-only by construction (no wallet actions). The library does not wrap it in a curated read API; each rule writes its own reads (`readContract`, `getBalance`, ...) against whatever ABI it needs, so custom rules are not boxed into a fixed set of helpers. The `blockNumber` carried in a bundle is not a write either: it pins the bucketized historical block every verifier reads balances at (every rule call passes `blockNumber: BigInt(ctx.blockNumber)`), so the tally is deterministic across clients (see "CRDT" and the bucket math in `chain/`).

So the only on-chain action a voter ever takes is acquiring the Pass, once, beforehand and unrelated to voting. Dropping the chain entirely is possible in principle — the gate and weight are a pluggable rule registry, so a non-chain gate rule could replace `erc721-min-balance` — but that would trade the Pass for some other Sybil-resistance mechanism, which v1 deliberately does not do.

## Why this resists the hard failure modes

### Can always-online peers drop votes?

No single peer can remove a vote from the network.

1. **Gossipsub floods.** Every honest peer re-forwards every valid message to its mesh. Declining to forward only affects a peer's own links; as long as one honest path connects you to the voter, you receive the vote.
2. **Aggregation is a monotonic union (a CRDT join).** Votes are signed and idempotent, so combining two peers' sets can only add, never remove. A liar can omit a vote from what it serves you but cannot subtract one that an honest peer serves.
3. **Cold start pulls from k peers and unions.** On join, fetch the current checkpoint from several independent peers via the libp2p fetch protocol and union them (union + LWW). To hide a vote, every source you consult must lack it.

The residual threat is an eclipse attack (all your peers are sybils), which is the standard p2p threat, not specific to voting. Mitigations are high peer count and diverse bootstrap/rendezvous.

### Does one Pass mean one vote?

The gate bounds Sybils **only because the v1 Pass is non-transferable.** The 5chan Pass is the MintPass — an SMS-gated proof-of-personhood ERC-721 (one phone ≈ one Pass), minted with a transfer cooldown, so a holder cannot cheaply rent it out or hand it to a briber. That non-transferability *is* the Sybil story; it belongs in the criteria comments so nobody later flips the Pass to freely transferable without realizing they have knocked the gate out.

The mechanism has a latent amplification that non-transferability is exactly what closes. Each bundle is verified at *its own* bucketized `blockNumber` and stays live for `voteExpiryBuckets`, and votes are keyed per **wallet** — so if a gating asset *could* move, one asset could back several concurrent live votes (held by wallet A at bucket N, transferred to B at N+1, and so on: each read is true at its own pinned block, none collapsed by LWW because the wallets differ). For the non-transferable Pass this cannot happen, but it **reopens** in two future configs, either of which must close it before shipping:

- a **transferable** gate asset (any ERC-721 that is not the MintPass);
- the reserved **`erc20-balance` weight** path — the same balance walked through many wallets, each bundle read at a different historical block, double-counts.

The structural fix, when one of those lands, is to key the LWW set by the **scarce asset (token id)** rather than the wallet, so a transfer *supersedes* the prior holder's vote instead of adding to it — which requires the gate rule to expose and prove *which* token backs a vote (reading token ids, not a bare `balanceOf`). A common per-contest snapshot block is the alternative, but it fights the heartbeat/expiry liveness model (every live vote is re-proven at its own recent block by design). Neither is in v1; pin a failing test vector alongside the first transferable or balance-weighted rule.

### Is there precedent for agreed state over pubsub?

Yes. A leaderless CRDT over pubsub — peers converging on a shared latest state with no leader — is what `go-ds-crdt` runs in production for IPFS Cluster (and what OrbitDB and Secure Scuttlebutt do). Those systems carry a Merkle-clock, whose extra property is that a head CID commits to its whole history, so *committed-ancestor* truncation is detectable. This design drops the Merkle-clock (see "CRDT"), giving up that marginal detectability — it only ever caught truncation of already-linked history, never omission of a vote nobody linked, which was always defended the same way regardless: **multi-peer checkpoint fetch + union**. The open-membership limit (you cannot learn about state nobody told you) is what that multi-peer fetch covers.

### Who publishes snapshots?

Anyone, trustlessly. A snapshot is a **checkpoint** — a root CID committing to a flat canonical array of the current winners (see "Checkpoints") — that every client re-verifies locally. The publisher has zero authority because state only ever joins by union, so a bad snapshot cannot subtract or forge. The expected availability layer is **bitsocial-seeder**: hosts run it, it joins the vote topics and helps peers cold-start (the eth-bootnode / IPFS-gateway pattern), but it is convenience, not an oracle — union-only means a seeder can never subtract or forge a vote. If all seeders vanish, live gossip plus multi-peer fetch still works, just with slower cold start.

### How fast is cold start?

For a topic with ~1000 votes, cold start is **fetch-bound, not compute-bound**. Each bundle is a small dag-cbor block (voting wallet address, one vote, blockNumber, one secp256k1 signature ≈ 0.3–1 KB), so ~1000 of them are well under ~1 MB total. Cold start fetches the current **checkpoint** — one root CID resolving to a handful of inlined chunk blocks (see "Checkpoints") — rather than chasing N blocks individually, so it is a few round trips, not N. The crypto is cheap (secp256k1 `ecrecover` is ~tens to low-hundreds of µs each, so recovering ~1000 vote signatures is still well under a second), and chain reads are lazy (tally only, only for ranking-relevant votes), so neither gates the join. The levers that shrink this further are the **checkpoint** format itself and **per-contest topics** (sync only the slot you care about).

### What happens when criteria change on upgrade?

A criteria change is a new rule `type` and/or new criteria bytes (including the `contestId`, which is part of the encoded object), which is a new CID, which is a **new topic**. Old clients that do not implement a `type` named in `requires` recuse themselves rather than miscount. The old topic drains as people update. There is no state migration because the heartbeat republish means there is no long-lived server state.

## Components

```
schema/        the wire and the criteria document (zod)
rules/  flat type -> rule registry (one kind; rule/weight slots)
chain/         historical-block reads (ERC-721 / ERC-20), chainTicker -> RPC, bucket math
verify/        the full validity pipeline (signature + constraints + gate + name) + verdict cache
crdt/          LWW winner-set per wallet + checkpoint reconciliation (monotonic union)
transport/     gossipsub topic: the async validate-before-forward gate (broadcast bundle CIDs + full-pipeline validator) + cold-start checkpoint fetch
tally/         deterministic per-contest aggregation over pre-validated bundles; lazy magnitude verify for balance-derived weight
```

The first three plus `verify/`, `crdt/`, and `tally/` have no libp2p dependency, so the engine is unit-testable without a network. libp2p appears only in `transport/`.

### Criteria document

Shipped static in the client bundle (offline), and `topic = "bitsocial-votes/" + CID(dag-cbor(criteria))`. dag-cbor is canonical (sorted keys, rejects `undefined`), so identical bytes yield an identical topic, which is the self-validating binding — and reordering keys does *not* fork the topic, only different values/fields/types do. Optionally also publish the bytes to IPFS for out-of-band verifiers.

**One criteria document describes one contest, so there is one topic per contest (directory slot).** The `contestId` value differs per document, which makes each contest's bytes distinct and forks the topic automatically. A client subscribes only to the contests it cares about, so cold start syncs one slot's votes rather than every directory's history — this is the main scaling lever (see "How fast is cold start?"). The `requires` block is the dependency manifest and doubles as version negotiation.

Because the rules live inside each document, a client that votes across a whole directory authors them in one manifest and derives one document per slot. [`5chan-directory-criteria.jsonc`](./5chan-directory-criteria.jsonc) is a **pure example** for 5chan — illustration only, not a shipped or authoritative config (the contract is the real 5chan Pass, but the bucket/expiry numbers, RPC URL, and the `/q/` override are placeholders). It is **not** a criteria document and is never encoded or published — it is a generator. It has a `defaults` block (every `CriteriaSchema` field except `name`/`contestId`: 5chan Pass gate via `erc721-min-balance` on Base, `constant` weight, the one-vote-per-topic bounds) and a `contests` list of all 63 directory slots (`/a/`, `/biz/`, `/g/`, …). The client derives each slot's document by shallow-merging its entry over the defaults — `criteria = { ...defaults, ...entry }` — canonically encodes it, and gets one topic per slot.

The point of the structure is that **rules are per-contest, not global**. `defaults` is only a DRY authoring convenience; it is flattened away at derivation, so there is no shared rule object at runtime — every contest is a complete, independent document and topic. Any entry may override any field (the override replaces that whole top-level field, no deep merge), so `/a/` and `/adv/` can run different gate rules, weight, vote bounds, or expiry. The example shows this on `/q/` (5chan Feedback), whose entry raises the gate to two Passes; in 5chan v1 every other slot just inherits the common Pass gate. Key order in the file is irrelevant since dag-cbor sorts canonically, so editing `defaults` re-forks every inheriting slot in lockstep, while editing one `contests` entry forks only that slot.

### Votes wire

```
VotesBundle { address, votes: Vote[], blockNumber, signature }
Vote        { board: { name?, publicKey }, vote }
```

The contest is not on the wire as a field: one topic decides one contest, so the contest is implied by the topic the bundle was published to (and bound into the signature — see below). A vote names the `board` and a numeric `vote`. A `board` is a pkc-js community identity — `{ name?, publicKey }`, where `publicKey` is the B58 IPNS name (e.g. `12D3KooW…`, the same shape pkc-js uses) and `name` is an optional **resolvable domain name** (e.g. `memes.bso`, resolved via [`@bitsocial/bso-resolver`](https://github.com/bitsocialnet/bso-resolver)'s `bitsocial` text record). A name is not a free label: the naming registry maps one name to one record at a time (only the name's owner can set it), so **a name is unique to one community — two communities cannot share a name**. The schema requires a name to be a dotted domain (it must carry a TLD) but deliberately does not pin the TLD — `.bso` is the current system and a future naming system may add others; what actually resolves is decided by the host-injected `nameResolvers` (see "Tally"). `publicKey` is validated strictly as a base58btc IPNS key (pkc-js's `isIpns` check) and a domain or non-decodable string is rejected — stricter than pkc-js's own loose `z.string().min(1)` field, because this is the vote-identity boundary.

**Board identity is `publicKey` alone, and a carried `name` is a verified claim on top of it.** The tally aggregates rows on `publicKey`, so a vote that omits `name` folds into the same row as one that carries it. But a carried name is never taken at face value: before aggregation the tally resolves it and **drops any bundle whose `name` does not resolve, or resolves to a different `publicKey` than the vote claims** — only bundles whose name↔publicKey pairing matches the registry are counted. Two votes carrying the same `name` with different `publicKey`s therefore cannot both survive: at most one matches the registry, and the other is rejected. Note `board.publicKey` is the *community's* identity, not the voter's — the voter is the gating-chain wallet in `address`. `address` is the voting wallet (the holder of the Pass / ERC-20) and MUST equal the address recovered from `signature`; it is carried so the LWW key and chain reads are available without re-recovering, and a forged `address` simply fails the recovery check.

The gating-chain wallet signs the bundle directly as **EIP-712 typed data** (`viem.signTypedData`; verification is `viem.verifyTypedData` / `recoverTypedDataAddress`). There is no separate author and no author→wallet binding — the recovered signer *is* the voter. The signed message binds:

- **the contest** — the criteria CID (equivalently the topic), carried as the raw binary CID bytes (see "Wire freeze"), so a signature can never be replayed onto another contest or another app's topic;
- **the `votes`** (each `board` `{ name, publicKey }` + numeric `vote`);
- **the `blockNumber`** — the LWW key and the bucketized block every verifier reads balances at.

The EIP-712 `domain` is `{ name: "bitsocial-votes", chainId }` with `chainId` set to the gating (`rule`) chain, which gives cross-chain/cross-app domain separation for free. There is deliberately no `version` field — nothing version-negotiates the domain, so a layout change is a new frozen vector, not a version bump. Constraints: `votes.length <= maxVotesPerAddress` (v1: 1, the one-vote-per-topic rule), each `vote` within `voteSchema` (v1: exactly 1), and the `votes` MUST name **pairwise-distinct `board.publicKey`s** — one entry per board, else the bundle is invalid. Without distinctness, `maxVotesPerAddress` bounds entries, not boards: under a future `maxVotesPerAddress: 5` criteria a wallet could list the same board five times and stack 5× weight on it, quietly turning the approval cap into a multiplier. The check is pure (bundle-local, no ordering, no chain) and criteria-independent — unlike the length cap and vote range, whose bounds live in the criteria — so `VotesBundleSchema` enforces it at parse time (see `src/schema/votes.ts`), before any signature or chain work. Distinct names need no rule of their own: the same name twice with different keys dies at name resolution (at most one pair matches the registry — see above), and with the same key it is a duplicate `publicKey`. An **empty `votes` array is always valid** — it is the withdrawal form (see "Cancelling a vote").

#### Wire freeze (v1)

The concrete EIP-712 `types` struct is frozen (see `src/signer/eip712.ts`, `BALLOT_TYPES`). Field names and order are part of the type hash, so any change is a breaking wire change that re-freezes the conformance vector:

```
Ballot { criteria: bytes, votes: Vote[], blockNumber: uint256 }
Vote   { board: Board, vote: int256 }
Board  { name: string, publicKey: string }
```

- **`criteria` is `bytes`, not a string.** It carries the raw binary CID (`cid.bytes`, 36 bytes for a CIDv1/dag-cbor/sha2-256 criteria — so `bytes32` cannot hold it). Hashing the canonical bytes avoids the multibase/version ambiguity a stringified CID would bake in, so independent clients recover identical signers. Note this is the CID of the whole criteria document, distinct from the `criteria.contestId` slot-code field inside it — the ballot field is named `criteria` (not `contestId`) to avoid that collision.
- **`vote` is `int256` (signed).** v1 is upvote-only, but signed leaves room for a future criteria to widen the range to downvotes without a layout change.
- **`board` is a `Board` struct `{ name, publicKey }`.** `publicKey` is the B58 IPNS community name; `name` is the optional resolvable domain. EIP-712 has no optional fields, so `name` is always signed as a string — the **empty string** when the wire vote carries no name. Since board identity is `publicKey` (not `name`), signing `""` versus a name for the same `publicKey` is the same board to the tally; but a non-empty `name` in the signature is a *claim* the tally verifies against the naming registry, and a bundle whose claim does not match resolution is dropped (see "Votes wire" and "Tally").
- **`blockNumber` is `uint256`.**
- The freeze is pinned by a fixed conformance vector (known key → known hash, signature, recovered signer) in `src/signer/eip712.test.ts`. Those literals are the cross-client spec: an independent implementation must reproduce them byte-for-byte.

### Cancelling a vote

There is no delete; the union only grows. A vote is removed two ways, both consistent with the monotonic union:

- **Active withdrawal.** Publish a newer bundle (higher `blockNumber`) with `votes: []`. LWW keyed by wallet picks the newest bundle, which now expresses *no vote*, so the tally drops it. The old bundle bytes may still be served, but `current()` resolves the wallet to the empty one. This is supersession, not deletion — the union still only gains a node. The empty **tombstone** is *not* re-signed (that would make it immortal); instead the scheduler **re-announces its existing CID** on the liveness cadence — re-broadcasting the winner CIDs without a fresh `blockNumber`, so the expiry clock is untouched — so a peer that missed its one-shot flood converges within a cycle. Once the tombstone expires the scheduler drops the intent and the bundle decays via expiry + prune. Correctness never depends on the re-announce: the superseded vote has a strictly lower `blockNumber`, so it decays strictly earlier regardless, and any stale copy a seeder replays is superseded by the retained tombstone until *it* expires. The re-announce only shortens how long a missed peer shows the withdrawn vote (from the vote's remaining life down to one cadence); the general fix is checkpoint cold-start (see "Checkpoints"). Passive withdrawal — dropping the intent without publishing an empty bundle (`PubsubVoter.forget`) — instead lets the vote lapse by its own expiry.
- **Passive expiry.** Stop republishing. A bundle is valid for `voteExpiryBuckets` after its `blockNumber` (the issue #25 heartbeat), so a wallet that goes silent has its vote decay on its own. Keeping a vote alive means periodically re-signing with a fresh `blockNumber`, on the **liveness cadence** `ceil(voteExpiryBuckets / 2)` buckets — half the expiry window, so one whole missed cycle still leaves slack before the vote decays (`republishIntervalBuckets` in the facade, derived per-contest from criteria; there is no global interval). This is deliberately a *liveness* cadence, coarser than the per-bucket block sampling the tally uses: a live vote is re-proven roughly twice per expiry window, not every bucket. Changing a vote is the same mechanism: a newer bundle naming a different board supersedes the old one.

### Lifecycle: one manifest, `start` / `stop` / `destroy`

A directory client owns one manifest and should not wire 63 contests by hand, so in v1 the manifest is a **required** `PubsubVoter` constructor option and the voter has a single lifecycle. The constructor derives every contest from that manifest (`deriveCriteria`) up front and keys them by their `contestId`; that id is how a host addresses one contest (`getContest({ contestId })`), so it must be **unique within the manifest** — a duplicate throws `DuplicateContestIdError`, and a missing manifest throws `MissingManifestError`. `start()` joins every derived contest (`VoteNetwork.start`) and arms the republish scheduler — one timer per contest on its `ceil(voteExpiryBuckets / 2)`-bucket liveness cadence, whose tick reloads the wallet's stored intent, re-signs it with a fresh `blockNumber`, and re-broadcasts. `stop()` clears the timers and leaves every topic but keeps the voter reusable; `destroy()` does the same and also disposes the vote store (closing its DB/file handles), so it is the discard path. There is no ad-hoc, manifest-free contest path in v1: a client that cares about a single contest passes a one-entry manifest (empty `defaults`, the criteria as the single `contests` entry) and reaches it with `getContest`. The scheduler and its re-sign tick land with the engine; the facade wires the options, the `start`/`stop`/`destroy` surface, and the timer teardown now.

### Persistence: the wallet's own vote intents

Republishing must re-sign *what this wallet chose*, so that choice has to survive a restart — but not as signed bundles. A bundle is immutable and pinned to a now-stale `blockNumber`, so re-serving an old one does nothing; the CRDT/blockstore already holds everyone's bundles by CID. What persists instead is the re-signable **intent**: `{ topic, address, votes, lastBucket }`, one per contest (`VoteStore` / `VoteIntent`, a small key-value seam mirroring the `BundleStore` seam). On `start()` the voter lists every intent and republishes it; every cast and every successful republish writes one back, and expiry/withdrawal deletes it. The library picks the backend by environment — **IndexedDB** in the browser, and on Node a **SQLite file under the constructor's `dataPath`** (a directory, one file inside it, WAL mode), the same `dataPath` convention pkc-js and `@bitsocial/bso-resolver` already take. There is no host-facing store seam: persistence is the library's own concern, configured by the single `dataPath` option (no injected store, no JSON-file backend, no env var). With no `dataPath` on Node the default is in-memory (intents lost on restart). The `VoteStore` contract stays an internal key-value interface so the voter can still be unit-tested against an in-memory store with no I/O.

### Identity: the voting wallet, nothing else (net-new; pkc-js has no signer for this)

There is no pkc-js author and no author→wallet binding. The thing that holds the Pass is an gating-chain wallet, and that wallet signs each vote directly (see "Votes wire"), so the only identity is the address recovered from the bundle's EIP-712 signature. Collapsing the old two-key model (ed25519 author + a secp256k1 binding vouching for it) into one wallet signature removes an entire verification layer — there is no binding to forge, and no separate binding timestamp to keep monotonic.

The old "sign once, delegate to a cheap key" rationale does not apply here: the client holds the wallet key and signs programmatically, so there is no per-vote human wallet prompt to avoid — paying one secp256k1 signature per bundle is free. Wallet-key compromise is a wallet-level concern outside the protocol (the same as for any on-chain asset); there is no protocol-level binding-revocation step because there is no binding. Supersession and replay are handled entirely by `blockNumber` LWW + bucket expiry (see "CRDT" and "Cancelling a vote").

### CRDT

Last-write-wins element-set keyed by the voting wallet address (recovered from the bundle's EIP-712 signature); value is the bundle; conflict resolution is highest `blockNumber`, tiebreak lowest bundle CID (for determinism across clients). That lowest-CID tiebreak *is* grindable — a signer can re-roll its own bundle's CID through the secp256k1 signature nonce — but harmlessly: it only ever orders a **single wallet's own** equal-`blockNumber` bundles, so the worst a wallet can do is pick which of *its own* duplicates wins, never move the cross-wallet ranking. This is why a CID tiebreak is safe *here* yet rejected for the tie **seed** (see "Tally"), which arbitrates between *different boards* and so must not take any creator-grindable input. Each bundle is a standalone dag-cbor block stored in the host's Helia blockstore — it carries no parent links. The live path gossips **individual bundle CIDs**; a peer fetches each unknown bundle by CID, verifies it, and LWW-merges it into its per-wallet winner-set. There is no DAG to walk and no notion of "heads." Eventual consistency comes from three overlapping mechanisms: gossipsub **flood** of each bundle CID; the **heartbeat re-gossip** that re-announces every live vote each `ceil(voteExpiryBuckets / 2)` liveness cycle (so a peer that missed a message catches it within a cycle); and periodic **checkpoint** reconciliation for cold-start and gap-filling (see "Checkpoints").

Expiry is a **read-time filter**: `current()` takes the current bucket and drops any bundle past `voteExpiryBuckets` after its `blockNumber`'s bucket, so a decayed vote never enters the tally. `prune` then drops expired and superseded-per-wallet bundles from the in-memory winner-set to bound memory; it is housekeeping, not the correctness guarantee, since the read-time filter already excludes anything it would remove. Because bundles carry no parent links, nothing forces an expired bundle to be retained — the next **checkpoint** simply omits it, which is what actually reclaims the storage the read-time filter only masks.

### Checkpoints

A **checkpoint** is a compacted snapshot of a topic's current state: the non-expired LWW winner per wallet (one bundle each), flattened, so a joining peer fetches one thing instead of chasing every bundle. It is the cold-start path and the compaction that bounds storage.

**Format.** A canonical **dag-cbor** array of the winning bundles, **sorted ascending by wallet address**, with the **full bundles inlined**. Because the array exceeds a single ~1 MiB block at any real size, it is split into chunks by a **size-cap fill** (append sorted bundles into a chunk until the next would exceed a fixed byte ceiling, then start a new chunk) and a small root manifest lists the chunk CIDs, yielding one **root CID**. This is a shallow, depth-2 *pagination* DAG — root → chunk blocks — **not** the history DAG we removed: it has no parent links, no causality, and is regenerated fresh each cut. There is **no `bucket` field**: each bundle self-describes via its `blockNumber`, and a peer derives the current bucket from the chain head, so a checkpoint needs no timestamp of its own.

**A sorted list, not a key-map.** The bundle already carries its `address` (which must equal the recovered signer), so a map keyed by address would duplicate it and invite a key-vs-value mismatch; and a map that shards across blocks needs a HAMT with pinned parameters, whereas a sorted list shards trivially by the size cap. In memory a peer indexes the decoded winners into a map anyway. (A HAMT keyed by address is the deferred variant for light clients that want a single-wallet lookup or absence proof without downloading the whole set; see "Future improvements".)

**Determinism → seeder agreement.** Canonical dag-cbor + the address sort + the size-cap rule make the bytes a pure function of the winner-set, so any two seeders with the same view produce the **same root CID** and their blocks dedupe. The exact byte layout (codec options, sort, chunk ceiling) is pinned by a fixed spec and a test vector. Gzip/tar are rejected (non-deterministic bytes → seeders disagree on the CID, and a payload of signatures/addresses/CIDs barely compresses); JSON is rejected (non-canonical key order/formatting loses that byte-identity).

**Publish and cold-join.** A seeder holds the root + chunk blocks (bitswap serves them) and announces the root CID (over the topic and/or the fetch responder). A joining peer subscribes; learns the root CID(s) from up to `k` independent peers; bitswaps the root + chunks (inlined, so no per-bundle round trip); recovers and checks every EIP-712 signature **offline**; **unions + LWW-merges** every checkpoint it pulled into one winner-set; does lazy chain reads only at tally time; and keeps merging live gossip on top. Divergent checkpoints from different peers are never "chosen between" — they are unioned, so a seeder that omits a vote cannot hide it as long as one honest peer includes it, and one that injects a forgery is caught by the local signature check. **No checkpoint proves completeness** — union across independent peers is the guarantee, not any single root CID.

**Expiry and withdrawals.** A checkpoint is cut from the current *non-expired* winners, so an expired vote is simply omitted at the next cut; its now-unreferenced block is GC-able. A winner that is an **empty-votes withdrawal** (see "Cancelling a vote") stays in the checkpoint until it *itself* expires: it is the tombstone that supersedes the wallet's earlier real vote under LWW, so dropping it early would let a union with a stale seeder resurrect the cancelled vote. It is safe to drop once expired, because its `blockNumber` exceeds the vote it superseded, so that original expired strictly earlier.

### Transport: gossipsub topic + validation

The transport is the only module that touches libp2p/helia. The host injects its running **Helia node** directly (the value `createHelia` returns) — no host-written adapter — and the transport drives that node's libp2p pubsub and blockstore itself. The node MUST carry a gossipsub service at `libp2p.services.pubsub` and a usable `blockstore`; `PubsubVoter`'s constructor validates both once via `requireHeliaServices` and throws `MissingPubsubError` / `MissingBlockstoreError` if either is absent or malformed (Helia's default services do not include pubsub), so the failure surfaces at construction rather than on the first `publish`/`subscribe`/`fetch`. (Bitswap itself is not a separately introspectable property — it is a block broker wired beneath `blockstore` — so the checkable guarantee is a well-formed blockstore, the surface bitswap retrieves through.) Over that node the transport does two things: **broadcast and receive individual bundle CIDs** on a gossipsub topic, and **fetch state** — bundles by CID through the blockstore, and the current **checkpoint** from peers on cold start. The topic is `"bitsocial-votes/" + CID(dag-cbor(criteria))` (see "Criteria document"), so subscribing to it is itself the proof that two peers ran identical rules.

Two fetch paths, by data kind:

- **Checkpoints are the cold-start pull** (a topic's compacted winner-set; see "Checkpoints"). On cold start the root CID is pulled from up to `k` peers via the libp2p **fetch protocol**, keyed by topic, and the checkpoints are unioned + LWW-merged (a single liar cannot hide a vote — see "Can always-online peers drop votes?").
- **Bundles are immutable, content-addressed blocks.** A bundle CID resolves to a `VotesBundle` — the actual signed votes — fetched **by CID via bitswap** through the injected Helia node's `blockstore`. There are no parent links and no ancestor walk: each gossiped CID is a standalone bundle, verified and LWW-merged on its own. A checkpoint answers "everything for this topic" in one root CID plus a handful of chunk fetches, so a seeder never serves the state block-by-block.

**Validation is an async gossipsub topic validator that runs the FULL validity pipeline before re-forwarding.** When the transport subscribes, it installs a validator for the topic — a function *resolving to* `accept | reject | ignore` that gossipsub runs on every received message *before re-forwarding it*. The validator is **async and returns a promise; gossipsub awaits it before `forwardMessage`**, so the whole fetch + verify pipeline runs before the message reaches the mesh. `reject` and `ignore` stop a message from propagating, and `reject` also lowers the sender's gossipsub peer score (with StrictSign, the penalty lands on the real origin peer). Implemented against `@libp2p/gossipsub`'s `topicValidators` map (`src/transport/gossip-validator.ts`, installed via `src/transport/helia.ts`).

Gating the *full* pipeline — not just a cheap byte check — is deliberate: **an invalid bundle must never be accept-and-forwarded.** Because every honest peer runs the same gate, an invalid bundle cannot cross even one honest hop, so it never reaches the mesh; the only propagation an attacker gets is to its own direct sybils. The accepted cost is that proving on-chain validity before forwarding means **every forwarding peer does one gate chain read per *unique* vote** (verdict-cached per bundle CID) — this retires the earlier "lazy tally only reads ranking-relevant votes" optimization *for validity*. Weight *magnitude* stays lazy at the tally (it is a ranking concern, not a validity one).

#### What stops someone publishing invalid bundle CIDs?

Nothing stops *publishing* them — and nothing needs to, because a bundle CID carries **no authority**. It is only a pointer; all trust comes from re-verifying the self-authenticating bundle behind it *before forwarding*. Every way a message can be bad is caught at the gate:

| Published thing | Caught by | Outcome |
|---|---|---|
| Malformed bytes / not a bounded list of CIDs / over size cap | gate, layer 1 (pre-fetch) | `reject` — not forwarded, sender scored down |
| Over the per-peer rate, or unfetchable within the timeout | gate (rate / fetch) | `ignore` — not forwarded, **no** penalty (not provably the sender's fault) |
| Bucketed **ahead of the verifier's chain head** (voter's head ahead of ours, clock skew, or an absurd `blockNumber`), or a board `name` that — resolved *at head* — does not resolve to the claimed `publicKey` | gate — freshness guard / name resolution, before forward | `ignore` — not forwarded, **no** penalty, and **not cached**: view-/clock-dependent, so two honest peers can legitimately disagree *right now* (see the `reject`-vs-`ignore` note below) |
| Resolves to a bundle that fails verification (bad signature, `address` ≠ recovered signer, vote out of range, over `maxVotesPerAddress`, duplicate `board.publicKey` within one bundle, **wallet the gate rejects at the bucket block**) | gate — offline verify + on-chain gate, all before forward | `reject` — not forwarded, sender scored down; never stored, re-served, or counted |
| Resolves to a valid, eligible, signed bundle | — | a legitimate vote: counted once per Pass-gated wallet (the NFT gate bounds Sybils **only because the v1 Pass is non-transferable — see "Does one Pass mean one vote?"**), LWW makes it deterministic, union makes it un-removable |
| Replayed/old bundle, or two conflicting bundles from one signer (equivocation) | LWW (higher `blockNumber`, tiebreak lower CID) + bucket expiry; the signature binds `blockNumber` so an old bundle cannot be re-stamped | resolves to one vote every client agrees on; a replayed older bundle is inert |
| A checkpoint or peer that omits known votes, or withholds | monotonic union + multi-peer cold-start fetch | cannot subtract a vote an honest peer serves (see "Can always-online peers drop votes?") |

The gate runs the pipeline **cheap-to-expensive with early exit**, so the costly steps run only for genuinely-new, signature-valid bundles, and a re-announced known bundle short-circuits on the verdict cache with zero network:

1. **Layer 1** (local, µs): decode to a bounded list of well-formed CIDs, within a size cap, within a per-peer rate. Malformed/over-cap → `reject`; over-rate → `ignore`.
2. **Fetch + freshness + offline verify** (per new bundle in the message): fetch the bundle by CID (bitswap; unfetchable within the timeout → `ignore`); a bundle whose bucket sample block is **ahead of the verifier's chain head** is not yet evaluable → `ignore` (uncached — it becomes evaluable once our head advances, and this also defuses an absurd-future `blockNumber` like `uint256.max`, which never merges); then recover the EIP-712 signature and check it matches `address`, and check the criteria-bound constraints (`maxVotesPerAddress`, vote range). Invalid → `reject`.
3. **On-chain gate + name resolution** (per new bundle): the gate `rule` must score `> 0n` at the bucket block — fail → `reject`, and a miss is remembered per `(wallet, sampleBlock)` so a flood of fresh-signed bundles from one ineligible wallet doesn't re-read the chain each time. Each `board.name` must then resolve through the host-injected `nameResolvers` to the claimed `publicKey` — fail → `ignore` (not `reject`): v1 resolves *at head*, so a re-point produces a window where honest peers disagree, and the verdict is not cached.

Only when every new bundle in the message verifies is it merged into the winner-set and the message forwarded (`accept`). The three verdicts are **who-gets-blamed**: `reject` for *provable* invalidity — a pure function of the bundle bytes and *pinned historical* chain state (bad signature, gate miss at the bucket block), so it is deterministic, safe to penalize the sender, and safe to cache; `ignore` for everything not provably the sender's fault — timeout / over-rate / unfetchable / already-seen, **plus the view-/clock-dependent cases** (a bundle bucketed ahead of our head, a `name` resolved at head during a re-point window) that two honest peers can disagree on right now. Using `reject` there would punish honest relayers for transient conditions and hand an attacker a grief vector — which is exactly why `ignore` exists alongside `reject`.

**Verdict-cache soundness.** The per-CID verdict cache stores only **terminal** verdicts — an `accept` or a provable `reject`. A transient `ignore` is never cached, because it can change as heads and naming records converge; caching it would pin a stale verdict (e.g. keep rejecting a name that has since re-pointed to match). Only bundle-bytes-plus-pinned-historical-state verdicts are stable enough to memoize.

The one residual is **resource exhaustion, not incorrectness**: a flood of plausible-looking but slow-to-reject CIDs costs wasted fetch/verify work. It is bounded by the per-message CID cap and size cap, per-peer rate limiting, a per-CID verdict cache (each unique bundle validated once), the **`(wallet, sampleBlock)` gate-miss negative cache** (a wallet already rejected at a pinned bucket block short-circuits the chain read for every further fresh-signed bundle it mints there), a `p-limit` concurrency cap on in-flight fetch/verify, gossipsub peer scoring (now effective for *semantic* badness, since `reject` fires pre-forward), and a bounded set of bundle CIDs per message — fetch only unknown ones, cap the count per message, and stop the moment one fails to verify. A whole-message deadline (v1: 10s) caps validator latency: on expiry the verdict is `ignore`, so a slow or hung fetch can never strand the message in gossipsub's mcache. The residual an attacker keeps is minting **fresh wallets** — each a genuine cache miss costing one gate read — but those bundles `reject` and score the *source peer* down, and public RPC endpoints self-limit; raising the cost of many fresh *peer IDs* is host-side gossipsub score tuning (see "Deferred pkc-js work").

### Tally

Deterministic per-contest aggregation over the CRDT's current bundles. Because the **forward-gate already verified every stored bundle** (signature, criteria constraints, on-chain gate, board-name resolution) before it was admitted, the tally does not re-do that work: it sums *weight magnitude* per board and orders the rows. Rows key on `board.publicKey`, so votes carrying different (registry-verified) names for the same key fold into one row. Implemented in `src/tally/tally.ts` over `crdt.current()`.

In v1 the weight rule is `constant` (1 pass = 1 vote), so the tally does **zero chain reads** — every read the system does is the gate's `rule` check. The **bound-based lazy verification** below is reserved for the balance-derived weight path (`erc20-balance`), where weight magnitude comes *from* a chain read and so needs a cheap ceiling before a full read:

The lever is that **weight verification only ever subtracts** — a vote's magnitude is its claimed ceiling or less, never more. So each board has a **ceiling** (sum of unverified claimed weight) and a **floor** (sum of verified weight); verifying raises the floor toward the ceiling. The winner is locked the moment the leader's floor ≥ every other board's ceiling. A full ranking locks pairwise top-down (`floor_i ≥ ceiling_{i+1}`); since 5chan only seats #1 per slot, the common case stops as soon as the leader is locked. For v1 `constant` weight every ceiling is trivially `1`, so claimed count *is* the ceiling and the sum is exact in one pass — the lazy machinery is inert.

**Name claims are resolved at the gate, before a bundle is ever stored** (see "Transport"): a vote's `board.name` never keys a row (`publicKey` does), but a carried name is a claim, resolved through the host-injected `nameResolvers` (the same `{ key, provider, resolve, canResolve }` interface pkc-js takes, implemented by `@bitsocial/bso-resolver` — injected on the `PubsubVoter` constructor, so this library imports no resolver), and **a bundle whose name does not resolve to the claimed `publicKey` is dropped before forwarding or storage**. So every bundle the tally sees already has truthful names, and two boards can never surface under one name — squatting a popular name with a freshly minted key produces bundles that never pass the gate. A vote may omit `name` entirely (a board with no registered name); it folds into its `publicKey` row. Key rotation needs no special case — when an owner re-points a name to a new `publicKey`, bundles signed for the old pair start failing the gate's resolution and decay out through the liveness heartbeat re-sign (`ceil(voteExpiryBuckets / 2)` buckets), whose fresh resolution picks up the new key. A re-point therefore fully takes effect within one republish cycle — up to half an expiry window.

**Ties.** An exact tie of verified weight is possible, so `floor_i ≥ ceiling_{i+1}` locks a rank only when strict *or* when board *i* also wins the tiebreak. Two tied boards are ordered by lowest `sha256(bucketBlockHash ‖ publicKey)` (byte comparison of digests; `bucketBlockHash` is the hash of the current bucket boundary block — head rounded down to `blocksPerBucket` — on the criteria's chain, read via `eth_getBlockByNumber` on the same `PublicClient` the tally already holds; `publicKey` is the board identity's raw bytes). The seed is deliberately **rolling**: any tie rule derived from values known at board creation is grindable, because keygen is free — a bare lowest-`publicKey` rule (or a fixed seed named in the criteria) lets a creator grind leading key bytes at creation time and buy every future tie — whereas a future bucket block hash cannot be predicted, so there is nothing to grind. Bundle-CID-based tiebreaks are rejected outright: secp256k1 signature nonces make a bundle's CID freely re-rollable by its own signer mid-contest (and heartbeat re-signing churns every CID once per republish cycle regardless of the exact cadence), so a CID is not a stable, ungrindable identity. Accepted costs: one extra RPC read per tally that actually hits a tie; a *persistent* tie re-rolls its winner each bucket (daily, at the example 43200-block Base buckets); and a transient disagreement window of seconds once per bucket, where two clients tallying right at the boundary disagree on which bucket is current until both have observed the boundary block — the same eventual consistency the CRDT accepts everywhere else. The boundary block lags the head by up to a full bucket, which also puts its hash past any realistic reorg depth. If multi-chain gating ever lands (see "Future improvements"), the seed chain must be picked deterministically — simplest is the first gate's chain; decide then.

The cheap-first ordering that used to live here (free signature check before a costly chain read) is now the **gate's** (see "Transport"): a bad signature drops a bundle for zero chain reads, before it is ever stored, so the tally only ever sees validity-checked bundles. For the reserved `erc20-balance` weight path the tally keeps its own DoS lever — a board whose ceiling is already below the leader's floor is never weight-read — but its magnitude has no free wire-side upper bound, so lazy tally there needs a separate bounding scheme (see "Open questions").

## Rules

Mirror the pkc-js challenge registry ([src/runtime/node/community/challenges/index.ts:94](https://github.com/pkcprotocol/pkc-js/blob/master/src/runtime/node/community/challenges/index.ts#L94)): a `Record<string, rule>` where user entries shadow builtins. Each rule owns its option schema and is evaluated at the bundle's bucket block via a `ChainClient`.

One flat registry, one file per `type` directly under `rules/`, each co-locating its option schema with its evaluation function — the pkc-js `{ type, options-validation, verify }` shape. `registry.ts` holds `builtinRegistry`, `resolveRegistry` (host overrides shadow built-ins by `type`), and `validateCriteriaRules` (the `rule`/`weight` refs must resolve and their options must parse, and every `requires.rules` name must resolve — else `UnknownRuleError` and the client recuses itself). `CriteriaSchema` stays loose (`{ type, ...options }`); the registry is the single source of truth for which `type`s are valid and how their options parse. Leaf rules depend only on the injected viem `PublicClient`, so they are unit-testable by stubbing its `readContract` (no network).

**One kind, one return type: `{ score: bigint }`.** There is a single rule kind; `evaluate` returns a result object whose non-negative `score` of `0n` means "does not qualify". (It is an object, not a bare `bigint`, so a slot-specific field — e.g. a self-declared `ceiling` for balance-derived weight — can be added later without another signature change; see "Open questions".) The criteria keeps two *slots* that draw from the one registry: the **rule** slot treats the score as a gate (`> 0n` admits, `0n` rejects), the **weight** slot treats it as the vote's magnitude. Final vote value = `evaluate(rule).score === 0n ? 0n : evaluate(weight).score`. A separate boolean gate kind was considered and rejected for the simplicity of one kind/one registry: a rule that needs a threshold (min Passes, min balance) bakes it in by returning `0n` below the threshold (`erc721-min-balance` returns the holding or 0n; `erc20-balance` takes an optional `min`). The same rule can therefore serve either slot — e.g. `erc721-min-balance` is a Pass gate in the `rule` slot and "1 vote per Pass" in weight. Each slot holds exactly one rule; combining several — an AND of gates, or an additive weight — is a deferred *structural* extension, not a combinator rule (see "Future improvements").

| type | typical slot | v1 | Purpose |
|---|---|---|---|
| `erc721-min-balance` | rule (gate) | yes | Hold at least N of an ERC-721 (5chan Pass); scores the holding or 0 |
| `constant` | weight | yes | Fixed score per wallet (1 pass = 1 vote) |
| `erc20-balance` | weight (or rule gate with `min`) | reserved | Score by ERC-20 balance (BSO) for the combo path (the README's weighted example) |

## Dependencies

Match pkc-js exact versions so dag-cbor blocks interop with the shared Helia 6 node: `zod 4.3.6`, `multiformats 13.4.2`, plus (added by the implementation later) `cborg 4.5.8`, `@ipld/dag-cbor`, `uint8arrays 5.1.0`. `viem 2.54.0` is a direct dependency: it is the chain client rules read through (`ctx.chain` is a viem `PublicClient`) and provides EIP-712 vote signing/verification (`signTypedData`, `verifyTypedData`, `recoverTypedDataAddress` — pkc-js has none of these). With the pkc-js author dropped, voting no longer needs ed25519 primitives. `@pkcprotocol/pkc-js`, `helia`, and `@libp2p/interface` are peer dependencies so libp2p is not double-bundled.

## Deferred pkc-js work

This library takes the host's Helia node directly and drives its gossipsub service and blockstore itself, so there is no host-written adapter to maintain — and the injected seam is exactly what pkc-js already holds, since `pkc.clients.libp2pJsClients[key]._helia` ([src/helia/libp2pjsClient.ts:20](https://github.com/pkcprotocol/pkc-js/blob/master/src/helia/libp2pjsClient.ts#L20)) *is* the Helia instance. It is private-by-convention and typed as raw Helia internals, though. Once this library's needs are firm, add a documented, version-stable accessor on pkc-js that returns that Helia node (carrying its gossipsub `libp2p.services.pubsub`, `blockstore`, and the registered-but-unexposed `fetch` service), rather than forcing consumers to reach through the churning `._helia` private field (for example the multiformats 13/14 split). pkc-js must register a gossipsub service on the node; without one, `PubsubVoter` construction throws `MissingPubsubError`. Track as a pkc-js issue and reference it here.

The forward-gate installs an **async** topic validator (a promise gossipsub awaits before `forwardMessage`), which needs no host-side gossipsub config — the pipeline simply does not resolve until it finishes. A fallback exists if that inline-await path is ever found to block gossipsub's message-processing loop unacceptably: switch to `asyncValidation: true` + `reportMessageValidationResult`, which requires the host to (a) construct gossipsub with `asyncValidation` enabled, (b) size the message cache (`mcacheLength` × `heartbeatInterval`) above the 10s validation budget so a slow validation is not evicted before it reports, and (c) tune `messageProcessingConcurrency`. Because those are gossipsub *constructor* options the host owns (this library only receives the running node), adopting the fallback is a documented pkc-js requirement, not a change this library can make alone. v1 uses the inline-await path and needs none of it. The cold-start **fetch protocol** for pulling a peer's current checkpoint (the `fetchWinnerCidsFromPeer` seam) is likewise still a pkc-js-side registration and is now **the** cold-start path — until it lands, cold start relies on live gossip plus the heartbeat re-gossip that re-announces every live vote each liveness cycle; the seam is wired but unset.

**Gossipsub peer-score tuning against fresh-peer-ID RPC floods.** The forward-gate's `reject` lowers a sender's score, but libp2p peer IDs are free, so an attacker minting fresh IDs can still make honest peers spend one gate chain read per *novel* wallet (bounded, not eliminated — see "Transport", resource-exhaustion residual). Identity cannot be made costly; *network position* can. The levers live in the gossipsub **constructor**, which the host owns (this library only receives the running node): weight the invalid-message penalty (P₄), enable the IP-colocation penalty (P₆) so many peer IDs behind one address are scored together, and set publish thresholds / a newcomer ramp so an unproven peer gets only a small budget until it delivers valid messages. Track as a pkc-js gossipsub-config item alongside the async-validation fallback above; none of it is a change this library can make alone.

## Future improvements

Combining several rules into one criteria slot is decided in principle but deferred: v1 (Pass gate + `constant` weight) and the weighted example (Pass gate + single-asset weight) each use exactly one rule per slot, so nothing ships a combinator yet. The direction is *structural* — mirroring pkc-js `settings.challenges`, a list the runtime iterates — not a general recursive combinator rule.

- **Gate combining — a `challenges`-style AND-array.** Make the `rule` slot `RuleRef[]`, every term required (each scores `> 0`), iterated by the runtime exactly as pkc-js ANDs its `challenges`. This expresses "hold a Pass **and** ≥ N BSO" with no combinator and no recursion. Building it: `CriteriaSchema.rule` becomes a non-empty array (normalized before canonical encoding, so one logical rule stays one topic), `validateCriteriaRules` iterates it, and `ChainReadContext` evolves from a single pre-resolved `chain` to a `chainFor(ticker)` resolver so each gate can name its own chain. An **OR**/nested gate, if ever needed, is a small *gate-only* `any` grouping decided then — still not a general combinator. (pkc-js also ORs via `exclude` indices, but every `exclude` dimension — `role`, `postScore`, `rateLimit`, account age — is a social/author signal absent for a bare voting wallet, so only the flat AND-array transfers.)
- **Weight combining — a weight-only, non-recursive reduction.** For multi-asset additive weight (e.g. `#Passes + BSO/1000`) add a reduction over weight-slot terms (a `sum`; `min`/`max` only if a real criteria needs them). It is weight-only — the gate has the canonical AND reduction, weight has none, which is exactly why weight must *name* its operator — and non-recursive (a flat list of leaves, not a tree of combinators). It also requires resolving the lazy-tally upper bound for balance-derived weight (see "Open questions"), since a summed balance carries no free wire-side ceiling.

Neither introduces a single "combinator" abstraction into the core: the gate composes by AND-listing, weight by a named reduction, and no rule recurses into the registry.

- **Chainless rules — the `chains` seam becomes optional, but the crash moves, it does not vanish.** A voting contest need not touch a blockchain (e.g. a `constant`-only rule), so the `chains: ChainClientFactory` seam should become optional, exactly as `signer?` already is. **Ownership decision:** whether a contest reads chain is a property of its *rule*, declared canonically by `criteria.requires.chains` (an empty record = chainless — it stays required-but-possibly-empty so intent is read from the topic-affecting criteria, not inferred from a runtime injection). The guard stays a *single* choke point, not a per-rule `if (!chain) throw`: every read already funnels through `#chainFor(ticker)` ([src/client/voter.ts, `#chainFor`](./src/client/voter.ts)), which throws when no client is configured. When `chains` is undefined, route that seam through a stub factory that throws one clear "no chains factory was injected" — chainless rules never call it and work; chain rules get one well-worded error in one place. But making the seam optional *alone* only relocates the failure: the constructor eagerly assumes a **gating chain** — it resolves `ruleTicker → ruleChain` and throws if absent, bakes `#chainId = requires.chains[ruleTicker].chainId` into **every EIP-712 ballot signature**, uses the gating chain as the tie-break block-hash seed, mandates `chain: ChainClient` in `ChainReadContext` (`src/rules/types.ts`), and `tickerForRef` throws on an empty `requires.chains` (`src/chain/ticker.ts`). So a genuinely chainless contest first needs three consensus-critical semantics decided: what `chainId` means in the EIP-712 `domain` with no chain, what seeds the tie-break, and what block (if any) a chainless rule is sampled at. **Sequencing:** do not loosen the type ahead of those answers — v1 is NFT-only (`erc721-min-balance` gate + `constant` weight, both chain-configured), so a `chains?` type would advertise a chainless path the constructor cannot honour, and a type that lies is worse than a required seam. Make `chains?` optional *and* make the gating-chain / `chainId` / tie-break path conditional-on-rule in the same change that ships the first chainless rule. This shares the "single pre-resolved chain → `chainFor` resolver" evolution the gate-combining bullet above already needs.

## Open questions

- **Lazy-tally upper bounds for non-constant weight.** Moved to [ROADMAP.md, Deferred](./ROADMAP.md#deferred-designed-not-shipped) — it is a precondition for shipping balance-derived weight, which is out of v1, so it lives with that deferred work rather than as a standing open question here.
- **Pinned-block name resolution — API gap closed upstream, multi-chain block-selection still open.** Board-name verification (see "Tally") should read the naming registry at a canonical historical block, exactly as balance reads are pinned to the bucket block. The resolver-API half is now solved upstream: [bso-resolver#3](https://github.com/bitsocialnet/bso-resolver/issues/3) is resolved (PR #4), so `@bitsocial/bso-resolver`'s `resolve` takes an optional `blockNumber?: bigint` (omitted ⇒ head) and the injected `NameResolver` seam here advertises it. But pinning was never *only* an API gap: the registry lives on its own chain (Ethereum for `.bso`) while the bucket boundary block is defined on the criteria's chain (Base in the 5chan example), so a deterministic read still needs a canonical per-bucket block *on the registry's chain* (per-chain bucket math, or mapping the bucket boundary timestamp to the nearest registry-chain block — the same multi-chain problem as the tie seed and the gate AND-array). That half remains open, so **v1 still resolves at head** (leaves `blockNumber` unset) and accepts a disagreement window around a re-point: two verifiers can transiently disagree about a name whose record just changed — the same eventual consistency the CRDT accepts elsewhere, bounded in practice because records change rarely and the heartbeat re-signs each liveness cycle (`ceil(voteExpiryBuckets / 2)` buckets), so the window is at most one republish cycle, up to half the expiry window, not a single bucket. Passing the pinned block is a one-line change once the registry-chain bucket block is chosen.
