# Design: @bitsocial/pubsub-votes

This document is the design of record, and the code implements it. Shipped: schemas, canonical encoding, topic derivation, the verify pipeline, LWW resolution, the tally, the transport's **validate-before-forward gossip gate** over **inline bundle deltas** (`src/transport/gossip-validator.ts`, `src/transport/messages.ts`), the binary bundle block encoding (`src/crdt/codec.ts`), the `PubsubVoter` / `Contest` / `ContestVote` reactive facade (`src/client/voter.ts`; republishing a live vote is the consuming client's job — see "Republishing is the client's job"), the forward-gate anti-amplification caches (`src/verify/gate-result-cache.ts`, `src/transport/accepted-dedup.ts`), and the full **root-record checkpoint sync** — on-demand encode (`src/checkpoint/codec.ts` + `rootRecord()`), the suppressed topic heartbeat, the libp2p-fetch responder/requester with the `MissingFetchError` construction guard, and the directed-bitswap divergence chase (`src/transport/chase.ts`). The **two-node gossipsub integration test** is now implemented too (`src/transport/integration/`, real `@libp2p/gossipsub` 16.0.3 over two loopback nodes, run via `npm run test:integration`); it surfaced and fixed a latent bug — the injected Helia node's `blockstore.get` is an async generator, not `Promise<Uint8Array>`, so the transport now normalises it via `adaptBlockstore` (`src/transport/helia.ts`). What remains is host-side: pkc-js adding gossipsub (>= 15.0.23; it currently ships 16.0.2) and `@libp2p/fetch` to the shared node's construction (see "Deferred pkc-js work"). This document explains what each part is for and why.

## Context

- **Consumer:** [5chan](https://github.com/bitsocialnet/5chan) and its [competitive directory system](https://github.com/bitsocialnet/5chan/blob/master/README.md#competitive-directory-system). Many communities compete for each directory slot; only the highest-voted community appears on the homepage. Current curation is manual pull requests to [`5chan-directories.json`](https://github.com/bitsocialnet/lists/blob/master/5chan-directories.json). This library replaces that with holder-decided, censorship-resistant voting.
- **Origin:** [pkc-js issue #25](https://github.com/pkcprotocol/pkc-js/issues/25), which sketched pubsub voting (settings in the topic, bucketized block numbers, per-bucket heartbeat republish). This design keeps the good parts and replaces the bare heartbeat with a CRDT — a grow-only LWW winner-set reconciled by checkpoints.
- **Host:** [pkc-js](https://github.com/pkcprotocol/pkc-js) (Public Key Communities) provides the shared libp2p/Helia node. It is *only* the node host: it supplies no identity and no signing for voting. This library adds everything chain-related, the CRDT, and the vote-signing/verification, none of which exist in pkc-js.

## Decisions

- **State model: grow-only LWW winner-set + checkpoints.** Votes form a last-write-wins element-set keyed by wallet (highest `blockNumber`, tiebreak lowest bundle CID). The live path gossips **individual bundle payloads as deltas** — each new or re-signed bundle inlined in the gossip message; each peer verifies the bundle from the message bytes (no block fetch) and LWW-merges it into its winner-set. Cold-start and compaction use **checkpoints** — a canonical flat array of the current winners, fetched from several peers and reconciled by union + LWW. Precedent for leaderless CRDT convergence over pubsub: OrbitDB, Secure Scuttlebutt, and Merkle-CRDTs / `go-ds-crdt` (Sanjuán et al., Protocol Labs 2020; powers IPFS Cluster's shared pinset with no leader) — though those sync via a Merkle-clock DAG where this design uses checkpoint reconciliation instead.
- **Weighting v1: 5chan Pass only.** ERC-721 ownership is the gate; weight is constant (1 pass = 1 vote). The rule design keeps a combo path open (Pass gate + BSO ERC-20 weight) without a redesign.
- **Voting is upvote-only in v1.** Downvotes turn a contest into a cheap weapon against competitors; approval-style upvotes (vote for as many communities as you like, capped by `maxVotesPerAddress`) avoid that. The wire keeps a numeric `vote` field so a future criteria could widen the range.
- **Identity is the voting wallet, not a pkc-js author.** The gating-chain wallet (whose private key 5chan or its clients hold, never pkc-js) signs each vote directly as EIP-712 typed data; the address recovered from that signature is the only identity in the system. There is no pkc-js author and no author→wallet binding — the signature *is* the identity. This drops a whole verification layer (no binding to forge or keep monotonic) and the ed25519 author signer.

## Votes are off-chain

A vote is never written to a blockchain. Casting a vote is: sign the bundle with the gating-chain wallet (EIP-712 typed data — no gas, an off-chain signature), store it as a dag-cbor bundle in the host's Helia blockstore (IPFS content-addressing, not a chain), and broadcast the new bundle's CID over gossipsub. There is no transaction and no gas.

The chain is **read-only**, and only to decide *who may vote* and *how much*:

- **Gate** (the `rule` slot, `erc721-min-balance`): does the wallet hold the 5chan Pass? This is the Sybil-resistance anchor. Without a scarce, hard-to-forge asset gating votes, anyone could generate unlimited identities and stuff the ballot.
- **Weight** (`constant` in v1, which reads no chain state at all; `erc20-balance` reserved for the combo path).

`ChainClient` is just a viem `PublicClient` — read-only by construction (no wallet actions). The library does not wrap it in a curated read API; each rule writes its own reads (`readContract`, `getBalance`, ...) against whatever ABI it needs, so custom rules are not boxed into a fixed set of helpers. The `blockNumber` carried in a bundle is not a write either: it pins the bucketized historical block every verifier reads balances at (every rule call passes `blockNumber: BigInt(ctx.blockNumber)`), so the tally is deterministic across clients (see "CRDT" and the bucket math in `chain/`).

So the only on-chain action a voter ever takes is acquiring the Pass, once, beforehand and unrelated to voting. Dropping the chain entirely is possible in principle — the gate and weight are a pluggable rule registry, so a non-chain gate rule could replace `erc721-min-balance` — but that would trade the Pass for some other Sybil-resistance mechanism, which v1 deliberately does not do.

## Why this resists the hard failure modes

### Can always-online peers drop votes?

No single peer can remove a vote from the network.

1. **Gossipsub floods.** Every honest peer re-forwards every valid message to its mesh. Declining to forward only affects a peer's own links; as long as one honest path connects you to the voter, you receive the vote.
2. **Aggregation is a monotonic union (a CRDT join).** Votes are signed and idempotent, so combining two peers' sets can only add, never remove. A liar can omit a vote from what it serves you but cannot subtract one that an honest peer serves.
3. **Cold start pulls from k peers and unions.** On join, fetch the current checkpoint from several independent peers via the libp2p fetch protocol and union them (union + LWW). To hide a vote, every source you consult must lack it.

The residual threat is an eclipse attack (all your peers are sybils), which is the standard p2p threat, not specific to voting. Mitigations are high peer count and diverse bootstrap/rendezvous.

### Does one Pass mean one vote?

The gate bounds Sybils **only because the v1 Pass is non-transferable.** The 5chan Pass is the MintPass — an SMS-gated proof-of-personhood ERC-721 (one phone ≈ one Pass), minted with a transfer cooldown, so a holder cannot cheaply rent it out or hand it to a briber. That non-transferability *is* the Sybil story; it belongs in the criteria comments so nobody later flips the Pass to freely transferable without realizing they have knocked the gate out.

The mechanism has a latent amplification that non-transferability is exactly what closes. Each bundle is verified at *its own* bucketized `blockNumber` and stays live for `voteExpiryBuckets`, and votes are keyed per **wallet** — so if a gating asset *could* move, one asset could back several concurrent live votes (held by wallet A at bucket N, transferred to B at N+1, and so on: each read is true at its own pinned block, none collapsed by LWW because the wallets differ). For the non-transferable Pass this cannot happen, but it **reopens** in two future configs, either of which must close it before shipping:

- a **transferable** gate asset (any ERC-721 that is not the MintPass);
- the reserved **`erc20-balance` weight** path — the same balance walked through many wallets, each bundle read at a different historical block, double-counts.

The structural fix, when one of those lands, is to key the LWW set by the **scarce asset (token id)** rather than the wallet, so a transfer *supersedes* the prior holder's vote instead of adding to it — which requires the gate rule to expose and prove *which* token backs a vote (reading token ids, not a bare `balanceOf`). A common per-contest snapshot block is the alternative, but it fights the heartbeat/expiry liveness model (every live vote is re-proven at its own recent block by design). Neither is in v1; pin a failing test vector alongside the first transferable or balance-weighted rule.

### Can valid votes clog the topic?

Two different worries hide under "clog", and they separate cleanly.

**Tally / state growth — no.** Votes are LWW-keyed by wallet, so one wallet occupies exactly **one** winner-set slot no matter how many bundles it publishes; superseded bundles decay by expiry + `prune`, and a checkpoint omits them (see "CRDT"/"Checkpoints"). Spamming valid votes therefore changes no outcome and grows no per-wallet state — the *incentive* to spam is already zero, and Sybil breadth is bounded by the non-transferable Pass ("Does one Pass mean one vote?").

**Bandwidth / CPU amplification — a bounded residual, handled locally.** An *eligible* wallet can re-sign the same vote with fresh signature nonces (or bump `blockNumber`), minting a distinct bundle each time. The per-CID verdict cache (keyed by the CID hashed from the bundle bytes) never hits on a fresh bundle, so absent other defences every forwarding peer would re-verify (one ecrecover + one gate chain read) and re-forward each one. Two local caches close this without any wire/criteria change:

- a **gate-result cache** keyed `(wallet, sampleBlock)` (`src/verify/gate-result-cache.ts`) memoizes the gate score — `0n` *or* `> 0n` — so an eligible wallet cycling choices within a bucket, or an ineligible wallet flooding fresh CIDs, costs **one** chain read per bucket, not one per bundle; and
- an **accepted-dedup** keyed `(wallet, bucket, votes)` (`src/transport/accepted-dedup.ts`) that makes the forward-gate return `ignore` for a same-bucket, same-choice re-sign — so the re-sign is not re-verified and, crucially, **not re-forwarded**. It is safe on the untrusted claimed `address` because a hit only ever *drops* a message (never accepts or stores), so it can suppress nothing an honest peer has not already forwarded. A genuine heartbeat re-sign lands in a *later* bucket and is not a duplicate.

Because `blockNumber` is bucketized and buckets are coarse (day-scale in the 5chan example), "a fresh vote per block" collapses into one bucket, so both caches see repeats. One further bound keeps the flood that *does* reach the gate from growing state: the **per-CID verdict cache is FIFO-bounded** (`src/verify/cache.ts`), so a flood of fresh, distinct bundles cannot grow memory without limit — an evicted terminal verdict only costs a re-verify on the next re-publish, never a wrong answer, and the gate-result cache still spares the chain read. (The per-fetch abort that used to guard the bitswap fetch slots is retired with the fetch itself — the bundle arrives inline, so there is no fetch pool to starve.) The remaining cost is a per-peer-rate-bounded decode + signature check, plus the fresh-peer-ID lever tracked in "Deferred pkc-js work".

**Why a per-wallet rate limit does NOT belong in the criteria.** A rate ("N per hour") is a *stateful, temporal* property across many messages, but a `rule` here must be a pure function of `(bundle, pinned historical chain state)` so that `reject` is provable and identical on every peer. Rate is not: two honest peers see different message subsets and timings, so they would disagree — it could only ever be an `ignore`-class *local* heuristic (which is exactly what the accepted-dedup is), never a consensus `reject`. Encoding it into the topic-deriving criteria bytes would give it authority it cannot honor, while LWW already removes the incentive it would be trying to price. So rate-limiting stays local and transport-layer, not a criteria knob.

### Is there precedent for agreed state over pubsub?

Yes. A leaderless CRDT over pubsub — peers converging on a shared latest state with no leader — is what `go-ds-crdt` runs in production for IPFS Cluster (and what OrbitDB and Secure Scuttlebutt do). Those systems carry a Merkle-clock, whose extra property is that a head CID commits to its whole history, so *committed-ancestor* truncation is detectable. This design drops the Merkle-clock (see "CRDT"), giving up that marginal detectability — it only ever caught truncation of already-linked history, never omission of a vote nobody linked, which was always defended the same way regardless: **multi-peer checkpoint fetch + union**. The open-membership limit (you cannot learn about state nobody told you) is what that multi-peer fetch covers.

### Who publishes snapshots?

Anyone, trustlessly. A snapshot is a **checkpoint** — a root CID committing to a flat canonical array of the current winners (see "Checkpoints") — that every client re-verifies locally. The publisher has zero authority because state only ever joins by union, so a bad snapshot cannot subtract or forge. The expected availability layer is **bitsocial-seeder**: hosts run it, it joins the vote topics and helps peers cold-start (the eth-bootnode / IPFS-gateway pattern), but it is convenience, not an oracle — union-only means a seeder can never subtract or forge a vote. If all seeders vanish, live gossip plus multi-peer fetch still works, just with slower cold start.

### How fast is cold start?

For a topic with ~1000 votes, cold start is **fetch-bound, not compute-bound**. Each bundle is a small dag-cbor block (voting wallet address, one vote, blockNumber, one secp256k1 signature ≈ 0.3–1 KB), so ~1000 of them are well under ~1 MB total. Cold start fetches the current **checkpoint** — one root CID resolving to a handful of inlined chunk blocks (see "Checkpoints") — rather than chasing N blocks individually, so it is a few round trips, not N. The crypto is cheap (secp256k1 `ecrecover` is ~tens to low-hundreds of µs each, so recovering ~1000 vote signatures is still well under a second), and the chain reads are **deferred and batched**: the join admits each pulled bundle on its offline checks alone and the per-wallet gate reads run in the background — grouped into one multicall per bucket sample block — so RPC latency never gates the first render (see "Background chain verification"). The levers that shrink this further are the **checkpoint** format itself and **per-contest topics** (sync only the slot you care about).

### What happens when criteria change on upgrade?

A criteria change is a new rule `type` and/or new criteria bytes (including the `contestId`, which is part of the encoded object), which is a new CID, which is a **new topic**. Old clients that do not implement a `type` named in `requires` recuse themselves rather than miscount. The old topic drains as people update. There is no state migration because the heartbeat republish means there is no long-lived server state.

## Components

```
schema/        the wire and the criteria document (zod)
rules/  flat type -> rule registry (one kind; rule/weight slots)
chain/         historical-block reads (ERC-721 / ERC-20), chainTicker -> RPC, bucket math
verify/        the full validity pipeline (signature + constraints + gate + name) + verdict cache
crdt/          LWW winner-set per wallet + checkpoint reconciliation (monotonic union)
transport/     gossipsub topic: the async validate-before-forward gate (inline bundle deltas + full-pipeline validator) + cold-start checkpoint pull (libp2p fetch)
tally/         deterministic per-contest aggregation over pre-validated bundles; lazy magnitude verify for balance-derived weight
```

The first three plus `verify/`, `crdt/`, and `tally/` have no libp2p dependency, so the engine is unit-testable without a network. libp2p appears only in `transport/`.

### Criteria document

Shipped static in the client bundle (offline), and `topic = "bitsocial-votes/" + CID(dag-cbor(criteria))`. dag-cbor is canonical (sorted keys, rejects `undefined`), so identical bytes yield an identical topic, which is the self-validating binding — and reordering keys does *not* fork the topic, only different values/fields/types do. Optionally also publish the bytes to IPFS for out-of-band verifiers.

**One criteria document describes one contest, so there is one topic per contest (directory slot).** The `contestId` value differs per document, which makes each contest's bytes distinct and forks the topic automatically. A client subscribes only to the contests it cares about, so cold start syncs one slot's votes rather than every directory's history — this is the main scaling lever (see "How fast is cold start?"). The `requires` block is the dependency manifest and doubles as version negotiation.

Because the rules live inside each document, a client that votes across a whole directory can author them from one local manifest and derive one document per slot. [`5chan-directory-criteria.jsonc`](./5chan-directory-criteria.jsonc) is a **pure example** for 5chan — illustration only, not a shipped or authoritative config (the contract is the real 5chan Pass, but the bucket/expiry numbers, RPC URL, and the `/q/` override are placeholders). It is **not** a criteria document and is never encoded or published — it is a generator. It has a `defaults` block (every `CriteriaSchema` field except `name`/`contestId`: 5chan Pass gate via `erc721-min-balance` on Base, `constant` weight, the one-vote-per-topic bounds) and a `contests` list of all 63 directory slots (`/a/`, `/biz/`, `/g/`, …). The app derives each slot's document by shallow-merging its entry over the defaults — `criteria = { ...defaults, ...entry }` — canonically encodes it, and gets one topic per slot. **The manifest is app-layer authoring, not library API**: the library only ever sees complete criteria documents (`createContest({ criteria })`), and because two clients must hold byte-identical documents to share a topic, the natural interchange between an operator and independent client implementations is the finished documents themselves, not a manifest plus a merge convention.

The point of the structure is that **rules are per-contest, not global**. `defaults` is only a DRY authoring convenience; it is flattened away at derivation, so there is no shared rule object at runtime — every contest is a complete, independent document and topic. Any entry may override any field (the override replaces that whole top-level field, no deep merge), so `/a/` and `/adv/` can run different gate rules, weight, vote bounds, or expiry. The example shows this on `/q/` (5chan Feedback), whose entry raises the gate to two Passes; in 5chan v1 every other slot just inherits the common Pass gate. Key order in the file is irrelevant since dag-cbor sorts canonically, so editing `defaults` re-forks every inheriting slot in lockstep, while editing one `contests` entry forks only that slot.

### Votes wire

```
VotesBundle { address, votes: Vote[], blockNumber, signature }
Vote        { community: { name?, publicKey }, vote }
```

The contest is not on the wire as a field: one topic decides one contest, so the contest is implied by the topic the bundle was published to (and bound into the signature — see below). A vote names the `community` and a numeric `vote`. A `community` is a pkc-js community identity — `{ name?, publicKey }`, where `publicKey` is the B58 IPNS name (e.g. `12D3KooW…`, the same shape pkc-js uses) and `name` is an optional **resolvable domain name** (e.g. `memes.bso`, resolved via [`@bitsocial/bso-resolver`](https://github.com/bitsocialnet/bso-resolver)'s `bitsocial` text record). A name is not a free label: the naming registry maps one name to one record at a time (only the name's owner can set it), so **a name is unique to one community — two communities cannot share a name**. The schema requires a name to be a dotted domain (it must carry a TLD) but deliberately does not pin the TLD — `.bso` is the current system and a future naming system may add others; what actually resolves is decided by the host-injected `nameResolvers` (see "Tally"). `publicKey` is validated strictly as a base58btc IPNS key (pkc-js's `isIpns` check) and a domain or non-decodable string is rejected — stricter than pkc-js's own loose `z.string().min(1)` field, because this is the vote-identity boundary.

**Community identity is `publicKey` alone, and a carried `name` is a verified claim on top of it.** The tally aggregates rows on `publicKey`, so a vote that omits `name` folds into the same row as one that carries it. But a carried name is never taken at face value: before aggregation the tally resolves it and **drops any bundle whose `name` does not resolve, or resolves to a different `publicKey` than the vote claims** — only bundles whose name↔publicKey pairing matches the registry are counted. Two votes carrying the same `name` with different `publicKey`s therefore cannot both survive: at most one matches the registry, and the other is rejected. Note `community.publicKey` is the *community's* identity, not the voter's — the voter is the gating-chain wallet in `address`. `address` is the voting wallet (the holder of the Pass / ERC-20) and MUST equal the address recovered from `signature`; it is carried so the LWW key and chain reads are available without re-recovering, and a forged `address` simply fails the recovery check.

The gating-chain wallet signs the bundle directly as **EIP-712 typed data** (`viem.signTypedData`; verification is `viem.verifyTypedData` / `recoverTypedDataAddress`). There is no separate author and no author→wallet binding — the recovered signer *is* the voter. The signed message binds:

- **the contest** — the criteria CID (equivalently the topic), carried as the raw binary CID bytes (see "Wire freeze"), so a signature can never be replayed onto another contest or another app's topic;
- **the `votes`** (each `community` `{ name, publicKey }` + numeric `vote`);
- **the `blockNumber`** — the LWW key and the bucketized block every verifier reads balances at.

The EIP-712 `domain` is `{ name: "bitsocial-votes", chainId }` with `chainId` set to the gating (`rule`) chain, which gives cross-chain/cross-app domain separation for free. There is deliberately no `version` field — nothing version-negotiates the domain, so a layout change is a new frozen vector, not a version bump. Constraints: `votes.length <= maxVotesPerAddress` (v1: 1, the one-vote-per-topic rule), each `vote` within `voteSchema` (v1: exactly 1), and the `votes` MUST name **pairwise-distinct `community.publicKey`s** — one entry per community, else the bundle is invalid. Without distinctness, `maxVotesPerAddress` bounds entries, not communities: under a future `maxVotesPerAddress: 5` criteria a wallet could list the same community five times and stack 5× weight on it, quietly turning the approval cap into a multiplier. The check is pure (bundle-local, no ordering, no chain) and criteria-independent — unlike the length cap and vote range, whose bounds live in the criteria — so `VotesBundleSchema` enforces it at parse time (see `src/schema/votes.ts`), before any signature or chain work. Distinct names need no rule of their own: the same name twice with different keys dies at name resolution (at most one pair matches the registry — see above), and with the same key it is a duplicate `publicKey`. An **empty `votes` array is always valid** — it is the withdrawal form (see "Cancelling a vote").

#### Wire freeze (v1)

The concrete EIP-712 `types` struct is frozen (see `src/signer/eip712.ts`, `BALLOT_TYPES`). Field names and order are part of the type hash, so any change is a breaking wire change that re-freezes the conformance vector:

```
Ballot { criteria: bytes, votes: Vote[], blockNumber: uint256 }
Vote   { community: Community, vote: int256 }
Community  { name: string, publicKey: string }
```

- **`criteria` is `bytes`, not a string.** It carries the raw binary CID (`cid.bytes`, 36 bytes for a CIDv1/dag-cbor/sha2-256 criteria — so `bytes32` cannot hold it). Hashing the canonical bytes avoids the multibase/version ambiguity a stringified CID would bake in, so independent clients recover identical signers. Note this is the CID of the whole criteria document, distinct from the `criteria.contestId` slot-code field inside it — the ballot field is named `criteria` (not `contestId`) to avoid that collision.
- **`vote` is `int256` (signed).** v1 is upvote-only, but signed leaves room for a future criteria to widen the range to downvotes without a layout change.
- **`community` is a `Community` struct `{ name, publicKey }`.** `publicKey` is the B58 IPNS community name; `name` is the optional resolvable domain. EIP-712 has no optional fields, so `name` is always signed as a string — the **empty string** when the wire vote carries no name. Since community identity is `publicKey` (not `name`), signing `""` versus a name for the same `publicKey` is the same community to the tally; but a non-empty `name` in the signature is a *claim* the tally verifies against the naming registry, and a bundle whose claim does not match resolution is dropped (see "Votes wire" and "Tally").
- **`blockNumber` is `uint256`.**
- The freeze is pinned by a fixed conformance vector (known key → known hash, signature, recovered signer) in `src/signer/eip712.test.ts`. Those literals are the cross-client spec: an independent implementation must reproduce them byte-for-byte.

#### Bundle wire encoding (size)

The dag-cbor **bundle block** (what the CID is over, what is stored and now inlined in the live gossip message) is a *separate* serialization from the EIP-712 signed struct above — changing it re-hashes the bundle CID but never the signature vector — so it is free to optimize for size. The lever is to carry the crypto material as **raw bytes, not strings**: `address` as 20 bytes (not a 42-char hex string), `signature` as its 65 raw bytes (not a 132-char hex string), and `community.publicKey` as the raw multihash bytes (not the ~52-char base58 string). Verification decodes each back to the string form the EIP-712 typed data expects (bytes → lowercase hex address, bytes → base58btc IPNS name), a lossless canonical mapping (casing is presentation — recovery compares case-insensitively), before `recoverTypedDataAddress`. Measured on the pinned vector this is **~30% smaller** (339 B → 235 B for a named one-vote bundle); the irreducible floor is the signature (65 B) + address (20 B) + `publicKey` (~38 B) + dag-cbor structure. Two things are deliberately *not* done: **short map keys** (a further ~15–20% at a real legibility/debuggability cost — the binary-field win is the clean one) and **compression** (rejected for the same reason as checkpoints — gzip is non-deterministic, breaking the byte-identity that lets seeders dedupe a bundle's CID, and signatures/keys barely compress). Dropping `address` entirely (it is recoverable from `signature`) saves 20 B but forces a recovery to learn the LWW key at read time; carrying it is the cheaper tradeoff and stays. **Where this matters:** a single ~200 B live delta flooding the mesh was already fine — the payoff is the **checkpoint**, where the same saving multiplies across ~1000 inlined winners (~120 KB off a ~500 KB snapshot).

### Cancelling a vote

There is no delete; the union only grows. A vote is removed two ways, both consistent with the monotonic union:

- **Active withdrawal.** Publish a newer bundle (higher `blockNumber`) with `votes: []` — `createContestVote({ criteria, votes: [] }).publish()`. LWW keyed by wallet picks the newest bundle, which now expresses *no vote*, so the tally drops it. The old bundle bytes may still be served, but `current()` resolves the wallet to the empty one. This is supersession, not deletion — the union still only gains a node. The withdrawal is broadcast **once**; a peer that missed the one-shot flood converges via checkpoint cold-start (see "Checkpoints"), and the tombstone's `blockNumber` strictly exceeds the vote it supersedes, so that vote decays strictly earlier regardless. This library no longer re-announces the tombstone on a timer — keeping a withdrawal alive to reach late peers, exactly like keeping any vote alive, is the client's concern (see "Republishing is the client's job").
- **Passive expiry.** Stop republishing. A bundle is valid for `voteExpiryBuckets` after its `blockNumber` (the issue #25 heartbeat), so a wallet that goes silent has its vote decay on its own. Keeping a vote alive means periodically re-publishing a freshly-signed bundle with a fresh `blockNumber` — which this library does **not** do automatically (see below). Changing a vote is the same mechanism: a newer bundle naming a different community supersedes the old one under LWW.

### Republishing is the client's job (not this library's)

A vote is not permanent: a bundle is valid only for `voteExpiryBuckets` after its `blockNumber`, so a live vote must be re-signed with a fresh `blockNumber` before it decays — the issue #25 heartbeat. **This library deliberately does not run that heartbeat.** It publishes a vote once, when the client calls `publish()`, and never re-signs or re-broadcasts on its own. Deciding *when* (or *whether*) to refresh a vote — every cycle, on app foreground, never — is the consuming client's concern (e.g. 5chan tracks which votes it holds and refreshes them on its own schedule). Baking an always-on background re-signer into a library that runs on the host's *shared* node was rejected as unintuitive: it would re-sign and re-broadcast on a timer the host cannot see, keep a wallet key hot indefinitely, and hide the vote's lifetime from the client that actually owns it. So there is no republish scheduler and no library-side persistence of vote intents (the old Node-SQLite / browser-IndexedDB store existed *only* to feed that scheduler across restarts, so it is gone too — the client keeps track of what it voted for).

What the library gives the client to make the refresh decision, all pure and chain-free:

- `republishIntervalBuckets(criteria)` — the recommended liveness cadence, `ceil(voteExpiryBuckets / 2)` buckets (half the expiry window, so one missed cycle still leaves slack before decay). A liveness cadence is deliberately coarser than the per-bucket block sampling the tally uses: a live vote need only be re-proven roughly twice per expiry window.
- the vote's own `blockNumber` (on the `VotesBundle` that `publish()` resolves) plus `criteria.voteExpiryBuckets` and `criteria.blocksPerBucket`: a vote sampled at bucket `b` expires once the current bucket exceeds `b + voteExpiryBuckets`.

To refresh, the client simply calls `createContestVote({ criteria, votes }).publish()` again — a new bundle at the current bucket supersedes the old one under LWW. Read-time expiry filtering (`current()`) and prune-on-read still drop decayed votes from every tally regardless, so a stale bundle a client forgot to refresh never miscounts; it just stops contributing once it expires.

### Facade: `PubsubVoter`, `Contest`, `ContestVote`

The public surface mirrors pkc-js/plebbit-js — a factory that mints small reactive objects, each with its own lifecycle and events. A contest is addressed by its **full criteria document**: `createContest({ criteria })` / `createContestVote({ criteria, votes })` strictly validate the document (`CriteriaSchema` — shape and canonical encodability — plus the rule registry, where an unimplemented rule throws `UnknownRuleError` and the client recuses) and derive its topic; the voter keys engines by that topic, so byte-identical criteria share one engine and byte-distinct criteria fork automatically — even two documents that happen to share a `contestId`, which is a byte-forking field inside the document, not a lookup key. There is no constructor-owned contest registry and no manifest option: how an app authors its documents (e.g. merging a local manifest, see "Criteria document") is app-layer, and the library only ever sees complete documents.

- **`PubsubVoter`** — the factory, holding the host-injected seams (`helia`, `chains`, `signer`, `nameResolvers`) once and owning **one engine per contest** (transport + CRDT + gate + tally, keyed by topic). It exposes `createContest`, `createContestVote`, and a `stop`/`destroy` lifecycle. There is no `start()` and no seeder-specific API: a full host or seeder participates in and serves the whole directory simply by creating + updating every contest, and the **checkpoint fetch responder registers itself lazily** — on the first engine's real topic join — and unregisters when the last engine leaves, so a node serves root records for exactly the contests it participates in, for exactly as long as it participates (symmetric with the heartbeat it broadcasts there). `stop()` leaves every topic but keeps the voter reusable (each contest can `update()` again); `destroy()` is **terminal** (mirroring pkc-js): it additionally marks the voter and its contests dead, so any later `createContest`/`createContestVote`, or a pre-existing `Contest.update()`/`ContestVote.publish()`, throws `VoterDestroyedError`. There is no store to dispose (republishing is the client's concern).
- **`Contest`** (read/subscribe; `createContest({ criteria })`) — one contest's reactive read view, the plebbit-js `subplebbit.update()` shape. `update()` joins the topic (cold-start pull + live gossip + heartbeat) and begins emitting; it fires an **`update`** event whenever incoming votes change the state — including each background check settlement, so a cold join renders a provisional tally immediately and refines as `chainVerified`/`nameResolved` flip (see "Background chain verification") — carrying a freshly recomputed tally readable as `contest.tally`, and an **`error`** event for a contest-level failure (the tally's chain read throws, or the background verifier hits an infra-class RPC/resolver failure and is retrying). The tally recompute is coalesced (a gossip burst yields one recompute, not one per bundle) and only runs while a `Contest` is subscribed, preserving the "idle contest does zero chain reads" property. `getTally()` remains for a forced fresh read; `stop()` leaves the topic.
- **`ContestVote`** (publish; `createContestVote({ criteria, votes })`) — one publishable ballot, the plebbit-js publication shape. `publish()` joins the topic if needed, signs the bundle for the current bucket, adds it to the local CRDT, and broadcasts it once as a live delta, resolving the `VotesBundle`. It emits **`publishingstatechange`** as its `publishingState` walks `stopped → signing → publishing → succeeded` (or `failed`), plus a terminal **`error`** event on failure. A withdrawal is `createContestVote({ criteria, votes: [] }).publish()`. On a voter built without a `signer`, `publish()` throws `ReadOnlyError`.

`createContest` returns the stable per-topic object (one CRDT/engine per topic per node — unlike plebbit, two live views of one contest cannot each hold an independent engine on a shared node), while each `createContestVote` is a fresh publication object over that shared engine. `ContestVote.publish()` and `Contest.update()` share one idempotent topic-join; engine teardown is owned by `Contest.stop()` / `PubsubVoter.stop()`.

### Identity: the voting wallet, nothing else (net-new; pkc-js has no signer for this)

There is no pkc-js author and no author→wallet binding. The thing that holds the Pass is an gating-chain wallet, and that wallet signs each vote directly (see "Votes wire"), so the only identity is the address recovered from the bundle's EIP-712 signature. Collapsing the old two-key model (ed25519 author + a secp256k1 binding vouching for it) into one wallet signature removes an entire verification layer — there is no binding to forge, and no separate binding timestamp to keep monotonic.

The old "sign once, delegate to a cheap key" rationale does not apply here: the client holds the wallet key and signs programmatically, so there is no per-vote human wallet prompt to avoid — paying one secp256k1 signature per bundle is free. Wallet-key compromise is a wallet-level concern outside the protocol (the same as for any on-chain asset); there is no protocol-level binding-revocation step because there is no binding. Supersession and replay are handled entirely by `blockNumber` LWW + bucket expiry (see "CRDT" and "Cancelling a vote").

### CRDT

Last-write-wins element-set keyed by the voting wallet address (recovered from the bundle's EIP-712 signature); value is the bundle; conflict resolution is highest `blockNumber`, tiebreak lowest bundle CID (for determinism across clients). That lowest-CID tiebreak *is* grindable — a signer can re-roll its own bundle's CID through the secp256k1 signature nonce — but harmlessly: it only ever orders a **single wallet's own** equal-`blockNumber` bundles, so the worst a wallet can do is pick which of *its own* duplicates wins, never move the cross-wallet ranking. This is why a CID tiebreak is safe *here* yet rejected for the tie **seed** (see "Tally"), which arbitrates between *different communities* and so must not take any creator-grindable input. Each bundle is a standalone dag-cbor block stored in the host's Helia blockstore — it carries no parent links. The live path gossips the **bundle payload itself as a delta** — the newly cast or re-signed bundle inlined in the gossip message, not a CID pointer — so a peer validates it straight from the message bytes (no fetch), stores it, and LWW-merges it into its per-wallet winner-set. Delivery inherits gossipsub's transitive mesh-flood reachability: a voter behind NAT or already offline still reaches every subscriber, because no peer needs to obtain the bundle from the origin — it is in the message. There is no DAG to walk and no notion of "heads." Eventual consistency comes from three overlapping mechanisms: gossipsub **flood** of each bundle payload; **client-driven re-gossip**, in which a wallet re-publishes **its own** live vote when its client decides to (this library does not do it automatically — see "Republishing is the client's job" — but the recommended cadence is `ceil(voteExpiryBuckets / 2)` buckets, so a peer that missed a still-live vote catches it within a cycle of the owner refreshing it); and periodic **checkpoint** reconciliation for cold-start and gap-filling (see "Checkpoints"). Note the shift from the CID-announce model: peers no longer re-flood the whole winner-set — a refreshed vote is just its own re-signed bundle, and cross-peer gap-filling for votes nobody is currently re-publishing leans on the checkpoint **root heartbeat + pull** (see "Checkpoints") rather than whole-set re-flooding. The checkpoint **root-record heartbeat** (the ~100 B advertisement, see "Checkpoints") is the one liveness signal this library *does* emit automatically — it carries no vote and keeps a converged topic near-silent, so it is safe to run on the shared node without the concerns that retired the vote re-signer.

Expiry is a **read-time filter**: `current()` takes the current bucket and drops any bundle past `voteExpiryBuckets` after its `blockNumber`'s bucket, so a decayed vote never enters the tally. `prune` then drops expired and superseded-per-wallet bundles from the in-memory winner-set to bound memory; it is housekeeping, not the correctness guarantee, since the read-time filter already excludes anything it would remove. Because bundles carry no parent links, nothing forces an expired bundle to be retained — the next **checkpoint** simply omits it, which is what actually reclaims the storage the read-time filter only masks.

### Checkpoints

A **checkpoint** is a compacted snapshot of a topic's current state: the non-expired LWW winner per wallet (one bundle each), flattened, so a joining peer fetches one thing instead of chasing every bundle. It is the cold-start / gap-fill pull and the compaction that bounds storage — and on the serve side it is **encoded on demand** from the responder's current winner-set (cached until the next merge), not cut on a schedule, so a joiner's view is as fresh as the responder's own state, not one cut-interval stale (see "On-demand encode" below).

**Format.** A canonical **dag-cbor** array of the winning bundles, **sorted ascending by wallet address**, with the **full bundles inlined**. Because the array exceeds a single ~1 MiB block at any real size, it is split into chunks by a **size-cap fill** (append sorted bundles into a chunk until the next would exceed a fixed byte ceiling, then start a new chunk) and a small root manifest lists the chunk CIDs, yielding one **root CID**. This is a shallow, depth-2 *pagination* DAG — root → chunk blocks — **not** the history DAG we removed: it has no parent links, no causality, and is regenerated fresh each cut. There is **no `bucket` field**: each bundle self-describes via its `blockNumber`, and a peer derives the current bucket from the chain head, so a checkpoint needs no timestamp of its own.

**A sorted list, not a key-map.** The bundle already carries its `address` (which must equal the recovered signer), so a map keyed by address would duplicate it and invite a key-vs-value mismatch; and a map that shards across blocks needs a HAMT with pinned parameters, whereas a sorted list shards trivially by the size cap. In memory a peer indexes the decoded winners into a map anyway. (A HAMT keyed by address is the deferred variant for light clients that want a single-wallet lookup or absence proof without downloading the whole set; see "Future improvements".)

**Determinism → seeder agreement.** Canonical dag-cbor + the address sort + the size-cap rule make the bytes a pure function of the winner-set, so any two seeders with the same view produce the **same root CID** *whenever asked* — no cut alignment needed. That agreement is what makes the k-peer pull cheap: a joiner first asks each peer only for its root CID, and when the roots match it downloads the payload **once**, knowing the others held identical bytes. The exact byte layout (codec options, sort, chunk ceiling) is pinned by a fixed spec and a test vector. Gzip/tar are rejected (non-deterministic bytes → seeders disagree on the CID, and a payload of signatures/addresses/CIDs barely compresses); JSON is rejected (non-canonical key order/formatting loses that byte-identity).

**The root record.** Discovery — "what is your current state?" — is answered by a tiny record, never by the payload blocks. Two shapes for the same state, matching the two transports (`src/transport/messages.ts`):

- The **heartbeat** (pushed on the topic) is the bare, constant-size advertisement `{ version: int, root: CID, count: int, sizeBytes: int }` in canonical dag-cbor (~100 B). `version` is for wire evolution; `count`/`sizeBytes` are **advisory hints** (how big a block pull would be, how many winners) — unverifiable until the blocks arrive, so nothing consensus-critical may depend on them. Constant size is the anti-amplification property: answering costs ~100 B whatever the state size — a payload-carrying response would hand a spammer a ~10,000× reflection (a ~60 B request per free peer ID compelling ~500 KB out of every seeder, with no validator or peer score in front of it).
- The **fetch-protocol response** (`FetchRootRecord`, pulled on cold join) additionally carries the **chunk-CID index** — the root manifest's contents, `{ ...advertisement, chunks: CID[] }`. This lets a cold joiner skip the root-manifest bitswap round-trip: it re-derives `CID(dag-cbor({ chunks }))`, checks it equals `root` (self-verifying, so the index is never a new trust vector — a lie fails the check and falls back to fetching the manifest block), and pulls **every chunk in parallel** instead of first fetching the manifest to learn their CIDs. The index is *not* the payload — it is an `O(sizeBytes / chunk-ceiling)` pointer list (~36 B per 1 MiB of checkpoint, the same size as the manifest block the joiner would otherwise fetch anyway), so a request still cannot compel more than a negligible bounded response; and because it rides only the *direct request/response* fetch protocol (never the broadcast heartbeat), the pubsub path keeps the stricter constant-size guarantee. This is the cold-join latency lever: the checkpoint pull drops from **two sequential** block round-trips (manifest → chunks) to **one** (chunks pulled in parallel).

The record is **deliberately unauthenticated — it carries zero authority**: the blocks self-verify against `root` (each hashes to its CID; the manifest links the chunks) and every bundle inside self-verifies via its EIP-712 signature, so a liar can only *omit* (defeated by union across peers) or point at garbage (fails local checks). No responder signature and no attestation of any kind (no "verified at block N", no claimed chain head): a field the requester must not rely on is pure attack surface, and an attestation anyone *did* rely on would be exactly the oracle this design forbids.

**Two transports for the same record — pull and push.** *Pull:* the **libp2p fetch protocol** (key → value, the same protocol IPNS-over-pubsub uses for record fetch) against peers already connected in the topic — a direct request/response stream, key `"bitsocial-votes/" + criteriaCid + "/root"` (the prefix only namespaces *which* contest a multi-contest seeder is being asked about; it is not a new pubsub topic). This is the on-demand path: cold join and reconnect-after-outage. The fetch service is a **required host service**: `PubsubVoter`'s constructor throws `MissingFetchError` when the injected node does not carry `libp2p.services.fetch`, exactly as it throws for a missing gossipsub (see "Transport" and "Deferred pkc-js work"). The pull **retries a thrown fetch until a deadline** (`#fetchRootWithRetry`, full-jittered exponential backoff): a client cold-loading a whole directory fires one fetch per contest at once, and the shared node's fetch handler defaults to only 32 concurrent inbound streams (libp2p 3.3.4's `DEFAULT_MAX_INBOUND_STREAMS`), so it *resets* the excess — and while more boards want to fetch than the node has slots the cap stays saturated, so a fixed handful of retries can lose the race and strand a board (measured: no retry → 32/63 boards converge; 5 fixed retries → 53/63; retry-to-deadline → **63/63 in ~6.3 s**). Only a throw retries — a definitive `undefined`/`null` ("no record") is returned as-is — and a torn-down contest abandons its retries. Two voter-level guards keep a directory join from creating that saturation *itself*: every cold-start fetch passes through a **voter-wide per-peer budget** (`COLD_START_PEER_FETCH_LIMIT`: at most 24 concurrent fetch streams to any one peer across ALL contests, so excess contests queue client-side instead of getting reset — the caps are enforced **per connection** per protocol per direction, so one connection's budget is exactly the scope of the remote's 32-inbound default, and other users of a shared seeder have their own connections and their own 32; the budget sits at 24 rather than 32 because running at the cliff still resets — our slot frees when the response lands but the responder decrements only on observing the stream *close*, and the same connection can carry fetch streams the budget cannot see, e.g. the host's IPNS-over-pubsub record fetches on the same protocol — the retry covers those residuals), and each contest asks a **random** `COLD_START_PEERS` of the topic's subscribers rather than deterministically the first listed, so a directory join spreads across the serving peers' independent stream caps (and budget queues) instead of funnelling every board through the same peer while the others idle. See "Deferred pkc-js work" for the optional host-side cap bump. *Push:* the same record is **heartbeated on the contest's existing gossipsub topic** — the IPNS-over-pubsub rebroadcast pattern, and its cadence too: **10 minutes** (the `go-libp2p-pubsub-router` rebroadcast default), jittered ±25% — so subscribed peers passively see each other's roots and a silently missed delta is detected without polling; the pubsub payload becomes a two-kind discriminated union (inline bundle delta, or root record), split at gate layer 1 (see "Transport"). Two rules keep the heartbeat quiet and unabusable, both improvements over IPNS's unconditional rebroadcast: **suppression** — a peer skips its jittered timer when it already heard a heartbeat matching its own root this interval, so a converged topic carries a few ~100 B records per interval no matter how many peers subscribe — and **at most one heartbeat per peer per interval regardless of stimuli**, so an attacker spraying divergent roots cannot make honest peers chorus in response. A peer that hears a root *differing* from its own publishes its own record (once, within its interval — so both sides learn of the split) and lazily chases the difference. Unlike an IPNS record, a root heartbeat is **not self-certifying**: it is an unverifiable *hint* about the publisher's own state (only *equality with your own root* is checkable), so the gate never `reject`s one on content — layer-1 shape + per-peer rate only — and acting on one is lazy and bounded.

**Block pull: directed bitswap at the peers that advertised the root.** The chunk blocks behind a root are fetched by CID via bitswap **from the connected peers whose record named that root** — they provably hold what they just advertised, so this is a directed transfer over existing connections, never provider-discovery toward an unreachable publisher (the failure mode that removed bitswap from the live path). When the fetch-protocol response supplied the **chunk index** (see "The root record"), the joiner has the chunk CIDs up front — verified against `root` locally — so it pulls all chunks **in parallel** and never fetches the root-manifest block at all; only a heartbeat-only divergence (no index) or an index that fails the local root check falls back to fetching the manifest first, then its chunks. Matching roots across peers make the pull cheap: download the DAG once, spreading chunk wants across the agreeing advertisers; a divergent root is pulled separately and unioned. Bulk serving rides bitswap's engine — per-peer ledgers, want-queue caps, requester-paced flow control — rather than a response burst a custom protocol would have to defend by itself. The chase side is bounded like everything else at the gate: a concurrency cap and a per-root deadline, so a spray of bogus roots queues bounded work; a bogus advertised root costs one directed attempt against its advertiser, and a peer whose roots repeatedly fail to resolve or verify is locally deprioritized.

**Cold-join pull.** A joining peer subscribes and asks up to `k` peers for their current root record over the fetch protocol (heartbeats also arrive passively, but the active ask gives an immediate answer and `k` independent ones). Peers come from **two discovery sources raced concurrently, neither blocking the other**: (1) the peers gossipsub already knows subscribe to this topic (`getSubscribers`), and (2) the **providers of the criteria CID from the host's HTTP content router(s)** — `libp2p.contentRouting.findProviders`, the pkc-js delegated-Routing-V1-over-HTTP pattern (no DHT), where the criteria CID doubles as the routing key so a provider record for it means "I run this contest." Source (2) is what makes a *fresh* join fast: on a cold start `getSubscribers` is typically empty until subscription gossip propagates (several seconds over a WAN link), whereas the router names a provider directly — so the peer dials it and fetches the root record **immediately**, without waiting to *learn* who is subscribed. It then pulls the blocks for each distinct root via directed bitswap from its advertisers; recovers and checks every EIP-712 signature **offline**; **unions + LWW-merges** every offline-valid bundle into one winner-set (admitted *provisionally* — the deferred gate reads and name resolutions run batched in the background and confirm or evict after the fact, see "Background chain verification"); and keeps merging live gossip on top. **Restart is just a cold join**: the winner-set is in-memory, so a restarted peer re-runs this pull — cheaply, since most blocks are already in its local blockstore. Root agreement across the `k` peers is a **download-once optimization, never an acceptance quorum**: when the roots match, pull the blocks once and know every advertiser held identical bytes; when they differ, pull *every* distinct root and union — a root served by only one peer is still pulled and merged, because trust is per-bundle (each signature self-verifies), not per-checkpoint (majority). A quorum-acceptance rule would hand a colluding majority of seeders exactly the vote-hiding power union exists to deny. Divergent checkpoints are never "chosen between" — they are unioned, so a seeder that omits a vote cannot hide it as long as one honest peer includes it, and one that injects a forgery is caught by the local signature check. **No checkpoint proves completeness** — union across independent peers is the guarantee, not any single root CID.

**On-demand encode; compaction on its own coarse cadence.** The serve side has **no cut cadence**: the fetch responder encodes its *current* non-expired winner-set when asked (the pure codec — `src/checkpoint/codec.ts`, `encodeCheckpoint`/`decodeCheckpoint`, no libp2p — is milliseconds for ~1000 winners), writes the blocks into the blockstore — content-addressed and idempotent, so directed bitswap can serve them (see "Block pull") — and caches the encoding until the next merge or expiry changes the set, so repeated pulls are free and a joiner's snapshot is seconds-stale at worst. One invalidation subtlety: the encode is async (blockstore writes), so the dirty flag is consumed in the same synchronous window as the winner-set snapshot, *before* the awaits — a state change landing mid-encode (typically a background-verifier settlement) re-dirties the flag and the next request re-encodes; clearing the flag after the encode would silently clobber that invalidation and pin a stale record until the next unrelated state change. The periodic-cut model this replaced was a bitswap-era artifact: a pre-stored, long-lived root only mattered when seeders announced roots for block-level dedup, and any cadence chosen for it traded freshness against churn — the earlier once-per-expiry-window cut would have left a joiner up to a full vote lifetime stale as a *freshness* mechanism. Determinism does not need the cadence: converged responders produce identical roots whenever asked (see "Determinism → seeder agreement"), and a vote landing mid-request only makes two responders transiently differ by that delta — which the union absorbs, costing one extra payload pull, never a wrong answer. What stays on a **coarse, bucket-aligned cadence** is **compaction**: the housekeeping that drops expired/superseded bundle blocks from the blockstore and persists a durable root across restarts — it rides the republish poll tick and never needed to be fresh. The responder also needs a **per-peer rate limit on snapshot requests** (same family as the gossip per-peer rate); with the encode cache, a request flood costs serving cached bytes, not re-encoding. The **consume** side — asking peers for their root and unioning — is a pkc-js fetch-service registration (see "Deferred pkc-js work").

**Expiry and withdrawals.** A checkpoint is cut from the current *non-expired* winners, so an expired vote is simply omitted at the next cut; its now-unreferenced block is GC-able. A winner that is an **empty-votes withdrawal** (see "Cancelling a vote") stays in the checkpoint until it *itself* expires: it is the tombstone that supersedes the wallet's earlier real vote under LWW, so dropping it early would let a union with a stale seeder resurrect the cancelled vote. It is safe to drop once expired, because its `blockNumber` exceeds the vote it superseded, so that original expired strictly earlier.

### Transport: gossipsub topic + validation

The transport is the only module that touches libp2p/helia. The host injects its running **Helia node** directly (the value `createHelia` returns) — no host-written adapter — and the transport drives that node's libp2p pubsub and blockstore itself. The node MUST carry a gossipsub service at `libp2p.services.pubsub` and a usable `blockstore`; `PubsubVoter`'s constructor validates both once via `requireHeliaServices` and throws `MissingPubsubError` / `MissingBlockstoreError` if either is absent or malformed (Helia's default services do not include pubsub), so the failure surfaces at construction rather than on the first `publish`/`subscribe`/`fetch`. The node MUST also carry the **libp2p fetch service** at `libp2p.services.fetch` — the root-record pull path (see "Checkpoints") — enforced the same way: `MissingFetchError` at construction. (The blockstore is still the checkable guarantee — it is where every validated bundle and every checkpoint block lives and where CIDs are computed. **Bitswap survives in exactly one role**: pulling checkpoint chunks from connected peers that advertised their root — a directed transfer from peers that provably hold the blocks — never on the live vote path and never provider-discovery toward an unreachable publisher.) Over that node the transport does three things: **broadcast and receive bundle payloads as live deltas** on a gossipsub topic — each new or re-signed bundle inlined so a receiver validates it straight from the bytes; **exchange root records** — pulled over the libp2p fetch protocol and heartbeated on the same topic; and **pull checkpoint blocks** via directed bitswap from root advertisers (see "Checkpoints"). The topic is `"bitsocial-votes/" + CID(dag-cbor(criteria))` (see "Criteria document"), so subscribing to it is itself the proof that two peers ran identical rules.

Two data kinds, two paths — and only one of them is a pull:

- **Live bundles ride inside the gossip message (no fetch).** A newly cast or re-signed bundle is inlined as canonical dag-cbor in the pubsub payload; the validator runs the full pipeline on those bytes, stores the block, and re-forwards. Delivery is the gossipsub mesh flood, so it needs no connectivity between receiver and voter — the guarantee bitswap could not give when the publisher is NAT'd or already offline. Each message carries **one bundle** (the wallet's own delta), not a winner-set, so its size is bounded and it does not re-flood state every heartbeat (see "Message size cap" and "CRDT"). This retires the per-CID bitswap fetch, the `GATE_FETCH_TIMEOUT_MS` per-fetch abort, and the fetch-slot `p-limit` from the live path.
- **Checkpoints are the cold-start / gap-fill sync** (a topic's compacted winner-set; see "Checkpoints"). The *root record* — tiny, constant-size, blockless — travels two ways: **pulled** on demand over the libp2p **fetch protocol** from peers already connected in the topic (cold join, reconnect), and **pushed** as a slow, suppressed **heartbeat on the same topic** (continuous divergence detection). The *blocks* behind a root are then fetched by CID via **directed bitswap at the peers that advertised it** — connected peers that provably hold what they just advertised, never provider-discovery toward an unreachable publisher (the failure mode that removed bitswap from the live path); the responder encodes its **current** winner-set on demand (see "Checkpoints"), so the snapshot is seconds-fresh, not cut-cadence-stale. The pulled checkpoints are unioned + LWW-merged (a single liar cannot hide a vote — see "Can always-online peers drop votes?").

#### Message size cap

Inlining the bundle means a pubsub message is now variable-size bytes, not a bounded count of fixed-size CIDs, so the transport bounds message size at two layers:

- **A hard transport ceiling** — gossipsub's own `maxTransmitSize`, generous but finite. Gossipsub drops an over-ceiling message before the validator runs, so a giant-message flood costs nothing. This is infrastructure, like an MTU — **not** a consensus value and **not** in the criteria. But it is a gossipsub **constructor** option, so the host owns it (this library only receives the running node): it belongs with the other host-side gossipsub tuning in "Deferred pkc-js work", not something this library can set. The library's own defence does not depend on it.
- **A semantic per-message cap** used for the layer-1 `reject` — the maximum size of one *canonical bundle valid under this criteria*. This is the library's own guard, enforced in the validator on `payload.length` before any decode/recover, so it holds regardless of what ceiling (if any) the host configured. (The other payload kind — the fixed-shape **root record**, see "Checkpoints" — is ~100 B by construction, so its cap is a trivial constant; the derived cap governs the bundle kind.) It is **derived**, not a free knob: `maxVotesPerAddress` (already in the criteria) × the max size of one vote entry (a fixed-length `publicKey`, a length-bounded `name`, a bounded `vote`) + the fixed `address`/`signature`/`blockNumber`/structure overhead. Every peer computes the identical bound from the identical criteria bytes, so an over-cap `reject` stays a pure function of the message bytes — deterministic, penalizable, and cacheable, exactly the `reject` class (unlike a rate limit, which is temporal and could only ever be a local `ignore`; see "Why a per-wallet rate limit does NOT belong in the criteria").

**Why the cap is derived, not a criteria field.** A raw `maxMessageBytes` in the criteria would duplicate bounds already implied by `maxVotesPerAddress` and could be set *below* what a legitimate bundle needs — self-inconsistent criteria that reject valid votes — while adding another topic-forking field for a number that should simply fall out of the others. Deriving it keeps the cap uniform across peers without a new authority it cannot honor. The one new *hygiene* bound this needs is a maximum `name` length in `VotesBundleSchema` (a fixed constant — **253 bytes, the DNS hostname limit**, decided — not a criteria knob; the same wire-shape category as the strict `publicKey` format check), so the per-vote-entry size — and thus the whole cap — is finite and derivable.

**Validation is an async gossipsub topic validator that runs the FULL validity pipeline before re-forwarding.** When the transport subscribes, it installs a validator for the topic — a function *resolving to* `accept | reject | ignore` that gossipsub runs on every received message *before re-forwarding it*. The validator is **async and returns a promise; gossipsub awaits it before `forwardMessage`**, so the whole verify pipeline (recover + constraints + on-chain gate + name resolution, all on the inlined bytes) runs before the message reaches the mesh. `reject` and `ignore` stop a message from propagating, and `reject` also lowers the sender's gossipsub peer score (with StrictSign, the penalty lands on the real origin peer). Implemented against `@libp2p/gossipsub`'s `topicValidators` map (`src/transport/gossip-validator.ts`, installed via `src/transport/helia.ts`).

Gating the *full* pipeline — not just a cheap byte check — is deliberate: **an invalid bundle must never be accept-and-forwarded.** Because every honest peer runs the same gate, an invalid bundle cannot cross even one honest hop, so it never reaches the mesh; the only propagation an attacker gets is to its own direct sybils. The accepted cost is that proving on-chain validity before forwarding means **every forwarding peer does one gate chain read per *unique* vote** (verdict-cached per bundle CID) — this retires the earlier "lazy tally only reads ranking-relevant votes" optimization *for validity*. Weight *magnitude* stays lazy at the tally (it is a ranking concern, not a validity one). This verify-before-anything ordering is a **live-path** requirement — gossipsub demands a verdict before it will forward. The cold-join chase has no forwarding decision to gate, so it admits on the offline checks and defers the chain work (see "Background chain verification"); what replaces the forwarding constraint there is the checkpoint rule "never re-serve what you have not verified".

#### What stops someone publishing invalid bundles?

Nothing stops *publishing* them — and nothing needs to, because a pubsub message carries **no authority**. All trust comes from re-verifying the self-authenticating bundle inlined in it *before forwarding*. Every way a message can be bad is caught at the gate:

| Published thing | Caught by | Outcome |
|---|---|---|
| Malformed bytes / not one canonical `VotesBundle` / over the derived size cap | gate, layer 1 (cheapest, pre-decode) | `reject` — not forwarded, sender scored down |
| Over the per-peer rate | gate (rate) | `ignore` — not forwarded, **no** penalty (not provably the sender's fault) |
| A well-formed **root-record heartbeat** within rate (see "Checkpoints") | gate, layer 1 (fixed shape) | `accept` — forwarded; an unverifiable *hint*, chased lazily via directed bitswap under a concurrency cap + deadline, never trusted |
| Bucketed **ahead of the verifier's chain head** (voter's head ahead of ours, clock skew, or an absurd `blockNumber`), or a community `name` that — resolved *at head* — does not resolve to the claimed `publicKey` | gate — freshness guard / name resolution, before forward | `ignore` — not forwarded, **no** penalty, and **not cached**: view-/clock-dependent, so two honest peers can legitimately disagree *right now* (see the `reject`-vs-`ignore` note below) |
| A bundle that fails verification (bad signature, `address` ≠ recovered signer, vote out of range, over `maxVotesPerAddress`, duplicate `community.publicKey` within one bundle, **wallet the gate rejects at the bucket block**) | gate — offline verify + on-chain gate, all before forward | `reject` — not forwarded, sender scored down; never stored, re-served, or counted |
| A valid, eligible, signed bundle | — | a legitimate vote: counted once per Pass-gated wallet (the NFT gate bounds Sybils **only because the v1 Pass is non-transferable — see "Does one Pass mean one vote?"**), LWW makes it deterministic, union makes it un-removable |
| Replayed/old bundle, or two conflicting bundles from one signer (equivocation) | LWW (higher `blockNumber`, tiebreak lower CID) + bucket expiry; the signature binds `blockNumber` so an old bundle cannot be re-stamped | resolves to one vote every client agrees on; a replayed older bundle is inert |
| A checkpoint or peer that omits known votes, or withholds | monotonic union + multi-peer cold-start fetch | cannot subtract a vote an honest peer serves (see "Can always-online peers drop votes?") |

The gate runs the pipeline **cheap-to-expensive with early exit**, so the costly steps run only for genuinely-new, signature-valid bundles, and a re-published known bundle short-circuits on the verdict cache with zero chain work:

1. **Layer 1** (local, µs): check the payload length against the derived per-message cap (see "Message size cap") and decode the two-kind discriminated union — **one** inline canonical `VotesBundle` delta, or a fixed-shape **root record** (see "Checkpoints"). Malformed/over-cap → `reject`; over the per-peer rate (each kind has its own limiter) → `ignore`. A well-formed, within-rate root record short-circuits here: `accept` (forwarded), handed to the chase logic as an unverifiable hint — steps 2–3 are bundle-only.
2. **Freshness + offline verify** (on the inlined bundle): hash the bytes to the bundle CID — this keys the verdict cache, so a re-published known bundle short-circuits here for the cost of one hash; a bundle whose bucket sample block is **ahead of the verifier's chain head** is not yet evaluable → `ignore` (uncached — it becomes evaluable once our head advances, and this also defuses an absurd-future `blockNumber` like `uint256.max`, which never merges); then recover the EIP-712 signature and check it matches `address`, and check the criteria-bound constraints (`maxVotesPerAddress`, vote range). Invalid → `reject`.
3. **On-chain gate + name resolution** (per new bundle): the gate `rule` must score `> 0n` at the bucket block — fail → `reject`, and a miss is remembered per `(wallet, sampleBlock)` so a flood of fresh-signed bundles from one ineligible wallet doesn't re-read the chain each time. Each `community.name` must then resolve through the host-injected `nameResolvers` to the claimed `publicKey` — fail → `ignore` (not `reject`): v1 resolves *at head*, so a re-point produces a window where honest peers disagree, and the verdict is not cached.

Only when the inlined bundle verifies is it merged into the winner-set and the message forwarded (`accept`). The three verdicts are **who-gets-blamed**: `reject` for *provable* invalidity — a pure function of the bundle bytes and *pinned historical* chain state (bad signature, gate miss at the bucket block), so it is deterministic, safe to penalize the sender, and safe to cache; `ignore` for everything not provably the sender's fault — timeout / over-rate / already-seen, **plus the view-/clock-dependent cases** (a bundle bucketed ahead of our head, a `name` resolved at head during a re-point window) that two honest peers can disagree on right now. Using `reject` there would punish honest relayers for transient conditions and hand an attacker a grief vector — which is exactly why `ignore` exists alongside `reject`.

**Verdict-cache soundness.** The per-CID verdict cache stores only **terminal** verdicts — an `accept` or a provable `reject`. A transient `ignore` is never cached, because it can change as heads and naming records converge; caching it would pin a stale verdict (e.g. keep rejecting a name that has since re-pointed to match). Only bundle-bytes-plus-pinned-historical-state verdicts are stable enough to memoize.

The one residual is **resource exhaustion, not incorrectness**: a flood of plausible-looking but slow-to-reject bundles costs wasted verify work. It is bounded by the derived per-message size cap (one bundle per message, so per-message work is capped by bytes), per-peer rate limiting, a **FIFO-bounded** per-CID verdict cache (each unique bundle validated once — the CID is hashed locally from the inline bytes, so a re-published known bundle costs one hash; the bound stops a fresh-bundle flood from growing the cache without limit — an evicted terminal verdict only costs a re-verify on re-publish), the **`(wallet, sampleBlock)` gate-result cache** (the score at a pinned bucket block is read once and reused — a `0n` miss short-circuits an ineligible wallet's flood, a `> 0n` hit short-circuits an *eligible* wallet re-signing or cycling choices within a bucket), the **`(wallet, bucket, votes)` accepted-dedup** (a same-bucket, same-choice re-sign is `ignore`d — not re-verified, not re-forwarded — so a re-sign flood does not amplify; see "Can valid votes clog the topic?"), a `p-limit` concurrency cap on in-flight validations (the gate chain read and name resolution are still network calls, so concurrent RPC work stays bounded), and gossipsub peer scoring (now effective for *semantic* badness, since `reject` fires pre-forward). A whole-message deadline (v1: 10s) caps validator *latency*: on expiry the verdict is `ignore`, so a slow or hung RPC or name resolution can never strand the message in gossipsub's mcache. (The old **per-fetch deadline** — `GATE_FETCH_TIMEOUT_MS` aborting a bitswap want so unfetchable CIDs could not starve the fetch pool, the kubo unbounded-wantlist DoS family — is retired with the fetch itself: the bundle arrives inline, so that attack surface no longer exists.) The residual an attacker keeps is minting **fresh wallets** — each a genuine cache miss costing one gate read — but those bundles `reject` and score the *source peer* down, and public RPC endpoints self-limit; raising the cost of many fresh *peer IDs* is host-side gossipsub score tuning (see "Deferred pkc-js work").

#### Transport constants (v1)

All **local, transport-layer knobs** — none is a consensus value, none appears in the criteria, none forks the topic; each can change per deployment without wire impact:

| Constant | v1 value | Why |
|---|---|---|
| Heartbeat interval | **10 min ± 25% jitter** | the IPNS-over-pubsub rebroadcast default (`go-libp2p-pubsub-router`); suppression + one-per-interval on top |
| Cold-join fan-out `k` | 4 | independent roots to union; more peers add bandwidth, not trust |

### Background chain verification

The cold-join chase admits a checkpoint's bundles in **two stages** (`src/transport/chase.ts` + `src/verify/background.ts`). Stage 1 runs **synchronously, before admit**: signature recovery + criteria constraints — local, µs each, so a forged or malformed bundle is never admitted at all. Stage 2 — the two *network* checks, the on-chain gate read and community-name resolution — is **deferred**: the bundle is admitted provisionally, the first tally renders immediately, and the background verifier confirms or evicts after the fact.

Why deferring is necessary: every unique wallet costs one gate read, and on a cold join those wallets are all new, so verifying inline would serialize N RPC round trips *before the first tally* — ~100 wallets against a real gateway is tens of seconds of latency the checkpoint fetch itself doesn't come close to. (The original benchmark never saw this because its fake chain answered in ~0 ms; the bench now runs the joiner against a mock ETH gateway — `benchmark/rpc-gateway.ts`, a real JSON-RPC server behind a real default-config viem client with a per-round-trip latency charge — and reports both `START→TALLY` and `START→VERIFIED`.) Deferring alone is not enough; the reads are also **batched**: rules may implement an optional `evaluateMany(walletAddresses, ctx)` (a transport optimization whose semantics MUST equal mapping `evaluate`), and `erc721-min-balance` implements it as one multicall3 `aggregate3` — a checkpoint's wallets share a bucket sample block, so N gate reads collapse into ~1 RPC round trip. Results feed the same `(wallet, sampleBlock)` gate-result cache and per-CID verdict cache the forward-gate uses, so nothing is ever read twice.

What keeps the provisional window honest:

- **The tally counts pending bundles but flags them** — one field per deferred operation, mirroring pkc-js's `nameResolved`: a row is `chainVerified` only once every contributing bundle's gate read confirmed, and `nameResolved` (present only when the row shows a name) once the shown name matched the registry. A leaderboard renders fast and refines; each settlement re-fires the coalesced `update` event. The synchronous stage never surfaces as a field — an admitted bundle has by construction passed it.
- **Never re-serve what you have not verified.** The checkpoint encoder reduces over *fully verified* bundles only, so a poisoned checkpoint cannot propagate ineligible votes through this node, and a hostile root cannot stuff what we serve. The eligibility-filtered LWW reduction falls back to a wallet's newest **verified** bundle while its newest overall is still pending — the wallet does not vanish from the served checkpoint mid-window.
- **Failed checks evict, with the gate's exact `reject`/`ignore` split**: a gate score of `0n` is a provable, deterministic fail → evict + cache the reject (a re-served copy short-circuits); a name that does not resolve to the claimed key is view-dependent → evict, uncached. Eviction restores the wallet's LWW predecessor: `prune` shields a superseded bundle while its superseder is provisional, exactly so a failing newcomer cannot destroy a verified vote.
- **Infra failures are nobody's verdict**: an RPC or resolver *throw* leaves the bundle pending (counted, flagged unverified), retries with capped full-jitter backoff, and surfaces through the Contest **`error`** event — so "the RPC is down" renders as a degraded, unverified tally instead of a silently quiet contest.
- **Own publishes take the same deferred path**: `publish()` admits this wallet's bundle provisionally and backgrounds its gate read, so an ineligible wallet's local tally converges to the network's rejection instead of silently disagreeing, and our own checkpoint never serves our own unverified vote.

The **live gossip path is unchanged** — gossipsub awaits the validator's verdict before forwarding, so the full pipeline (including the chain read) still runs inline there, where it is amortized to one read per unique `(wallet, bucket)` by the caches and never batches N-at-once anyway.

### Persistent caches (`dataPath`)

Two of the engine's caches persist across restarts, under the voter's `dataPath` (`src/storage/` — the pkc-js storage stack, deliberately: better-sqlite3 per-cache databases under `{dataPath}/lru-storage/` on Node, localforage/IndexedDB with pkc-js's dual-instance round-robin LRU in the browser via a package.json `browser`-field remap of the one storage module, in-memory for `dataPath: false`). Both are shared across every contest on the voter. Everything else — verdict cache, accepted-dedup, rate windows, tally/checkpoint memoization — is session state by design and stays in memory.

- **Gate results** (`(ruleHash, wallet, sampleBlock) → score`, `src/verify/gate-result-cache.ts`). The score is a pure function of pinned historical chain state, so persisting it is free of staleness by construction: a restart re-serves every settled gate read from the store instead of the RPC, which is what moves `START→ALL-VERIFIED` on a warm re-join. The `ruleHash` (sha256 of the canonical dag-cbor `criteria.rule` + chainId) is exactly the sharing boundary — two contests over one gate (a 5chan-style directory of boards on the same Pass) share each other's reads; different gates cannot collide. Caching sits at this *semantic* layer rather than raw RPC because the background verifier batches through multicall3: batched calldata varies with batch composition, so RPC-level keys would rarely repeat. Eviction is deterministic, not heuristic: a score at bucket B is only ever consulted while bundles from B are admissible, so entries older than the oldest admissible sample block are provably dead and purged per contest whenever a head read advances the expiry boundary (the store's LRU bound is only the backstop). An in-memory FIFO front keeps the verify hot path off the store; a broken store read or write degrades to a live chain read, never an error.
- **Name resolutions** (`src/verify/name-resolution-cache.ts`) — a port of pkc-js's `NameResolutionCache`, same rule so a host reasons about one policy across both libraries: LRU-bounded (5000, pkc-js's cap), **no stored TTL**, freshness is a per-call max-age (`Cache-Control: max-age` semantics), only successful resolutions are stored (a failure is retried live, never negatively cached), keyed `{name}::{resolverKey}::{sha256(provider)}`. Both verify call sites (inline forward-gate and background verifier) read with **max-age 3600s** — pkc-js's own background-resolution value — so the RPC cost of popular names is bounded while a re-pointed name is honored within at most an hour, strictly tighter than the republish-cycle window the re-point analysis in "Tally" already accepts. This cache exists here because pkc-js caches at *its* call sites, not inside the injected resolver — without it every verify pays a live registry read pkc-js would have served from cache. v1 resolves at head; once pinned-block resolution lands (see "Open questions"), a resolution becomes deterministic like a gate read and the max-age can go.

### Tally

Deterministic per-contest aggregation over the CRDT's current bundles. The tally never re-does verification: every aggregated bundle passed the synchronous offline checks at admission, and each entry carries its deferred-check state (`BundleChecks` — settled inline by the forward-gate for live gossip, settled in the background for cold-join admits, see "Background chain verification"). The tally sums *weight magnitude* per community, folds the check states into each row's **`chainVerified`** / **`nameResolved`** flags (a row is `chainVerified` only when every contribution's gate read confirmed; `nameResolved` exists only when the row shows a name, and a resolved name is preferred over a pending one for the display slot), and orders the rows. Rows key on `community.publicKey`, so votes carrying different names for the same key fold into one row. Implemented in `src/tally/tally.ts` over `crdt.currentEntries()`.

In v1 the weight rule is `constant` (1 pass = 1 vote), so the tally does **zero chain reads** — every read the system does is the gate's `rule` check. The **bound-based lazy verification** below is reserved for the balance-derived weight path (`erc20-balance`), where weight magnitude comes *from* a chain read and so needs a cheap ceiling before a full read:

The lever is that **weight verification only ever subtracts** — a vote's magnitude is its claimed ceiling or less, never more. So each community has a **ceiling** (sum of unverified claimed weight) and a **floor** (sum of verified weight); verifying raises the floor toward the ceiling. The winner is locked the moment the leader's floor ≥ every other community's ceiling. A full ranking locks pairwise top-down (`floor_i ≥ ceiling_{i+1}`); since 5chan only seats #1 per slot, the common case stops as soon as the leader is locked. For v1 `constant` weight every ceiling is trivially `1`, so claimed count *is* the ceiling and the sum is exact in one pass — the lazy machinery is inert.

**Name claims are verified before they can mislead**: a vote's `community.name` never keys a row (`publicKey` does), but a carried name is a claim, resolved through the host-injected `nameResolvers` (the same `{ key, provider, resolve, canResolve }` interface pkc-js takes, implemented by `@bitsocial/bso-resolver` — injected on the `PubsubVoter` constructor, so this library imports no resolver). On the live path the gate resolves it **before forwarding or storage**; on the cold-join path resolution is deferred to the background verifier, during which the row honestly reports `nameResolved: false` — and **a name that resolves to a different `publicKey` than claimed evicts the bundle**, so a mismatched name is never shown as resolved and two communities can never surface under one name: squatting a popular name with a freshly minted key produces bundles that never settle as verified anywhere. A vote may omit `name` entirely (a community with no registered name); it folds into its `publicKey` row. Key rotation needs no special case — when an owner re-points a name to a new `publicKey`, bundles signed for the old pair start failing the gate's resolution and decay out by expiry; a voter's client that refreshes on the recommended `ceil(voteExpiryBuckets / 2)`-bucket cadence re-signs at head, whose fresh resolution picks up the new key. So a re-point takes full effect within one republish cycle for clients that refresh (up to half an expiry window), and by plain expiry for those that do not — this library does not re-sign on its own (see "Republishing is the client's job").

**Ties.** An exact tie of verified weight is possible, so `floor_i ≥ ceiling_{i+1}` locks a rank only when strict *or* when community *i* also wins the tiebreak. Two tied communities are ordered by lowest `sha256(bucketBlockHash ‖ publicKey)` (byte comparison of digests; `bucketBlockHash` is the hash of the current bucket boundary block — head rounded down to `blocksPerBucket` — on the criteria's chain, read via `eth_getBlockByNumber` on the same `PublicClient` the tally already holds; `publicKey` is the community identity's raw bytes). The seed is deliberately **rolling**: any tie rule derived from values known at community creation is grindable, because keygen is free — a bare lowest-`publicKey` rule (or a fixed seed named in the criteria) lets a creator grind leading key bytes at creation time and buy every future tie — whereas a future bucket block hash cannot be predicted, so there is nothing to grind. Bundle-CID-based tiebreaks are rejected outright: secp256k1 signature nonces make a bundle's CID freely re-rollable by its own signer mid-contest (and heartbeat re-signing churns every CID once per republish cycle regardless of the exact cadence), so a CID is not a stable, ungrindable identity. Accepted costs: one extra RPC read per tally that actually hits a tie; a *persistent* tie re-rolls its winner each bucket (daily, at the example 43200-block Base buckets); and a transient disagreement window of seconds once per bucket, where two clients tallying right at the boundary disagree on which bucket is current until both have observed the boundary block — the same eventual consistency the CRDT accepts everywhere else. The boundary block lags the head by up to a full bucket, which also puts its hash past any realistic reorg depth. If multi-chain gating ever lands (see "Future improvements"), the seed chain must be picked deterministically — simplest is the first gate's chain; decide then.

The cheap-first ordering that used to live here (free signature check before a costly chain read) is now the **gate's** (see "Transport"): a bad signature drops a bundle for zero chain reads, before it is ever stored, so the tally only ever sees validity-checked bundles. For the reserved `erc20-balance` weight path the tally keeps its own DoS lever — a community whose ceiling is already below the leader's floor is never weight-read — but its magnitude has no free wire-side upper bound, so lazy tally there needs a separate bounding scheme (see "Open questions").

## Rules

Mirror the pkc-js challenge registry ([src/runtime/node/community/challenges/index.ts:94](https://github.com/pkcprotocol/pkc-js/blob/master/src/runtime/node/community/challenges/index.ts#L94)): a `Record<string, rule>` where user entries shadow builtins. Each rule owns its option schema and is evaluated at the bundle's bucket block via a `ChainClient`.

One flat registry, one file per `type` directly under `rules/`, each co-locating its option schema with its evaluation function — the pkc-js `{ type, options-validation, verify }` shape. `registry.ts` holds `builtinRegistry`, `resolveRegistry` (host overrides shadow built-ins by `type`), and `validateCriteriaRules` (the `rule`/`weight` refs must resolve and their options must parse, and every `requires.rules` name must resolve — else `UnknownRuleError` and the client recuses itself). `CriteriaSchema` stays loose (`{ type, ...options }`); the registry is the single source of truth for which `type`s are valid and how their options parse. Leaf rules depend only on the injected viem `PublicClient`, so they are unit-testable by stubbing its `readContract` (no network).

**One kind, one return type: `{ score: bigint }`.** There is a single rule kind; `evaluate` returns a result object whose non-negative `score` of `0n` means "does not qualify". (It is an object, not a bare `bigint`, so a slot-specific field — e.g. a self-declared `ceiling` for balance-derived weight — can be added later without another signature change; see "Open questions".) The criteria keeps two *slots* that draw from the one registry: the **rule** slot treats the score as a gate (`> 0n` admits, `0n` rejects), the **weight** slot treats it as the vote's magnitude. Final vote value = `evaluate(rule).score === 0n ? 0n : evaluate(weight).score`. A separate boolean gate kind was considered and rejected for the simplicity of one kind/one registry: a rule that needs a threshold (min Passes, min balance) bakes it in by returning `0n` below the threshold (`erc721-min-balance` returns the holding or 0n; `erc20-balance` takes an optional `min`). The same rule can therefore serve either slot — e.g. `erc721-min-balance` is a Pass gate in the `rule` slot and "1 vote per Pass" in weight. Each slot holds exactly one rule; combining several — an AND of gates, or an additive weight — is a deferred *structural* extension, not a combinator rule (see "Future improvements").

| type | typical slot | v1 | Purpose |
|---|---|---|---|
| `erc721-min-balance` | rule (gate) | yes | Hold at least N of an ERC-721 (5chan Pass); scores the holding or 0 |
| `constant` | weight | yes | Fixed score per wallet (1 pass = 1 vote) |
| `erc20-balance` | weight (or rule gate with `min`) | reserved | Score by ERC-20 balance (BSO) for the combo path (the README's weighted example) |

## Deferred pkc-js work

This library takes the host's Helia node directly and drives its gossipsub service and blockstore itself, so there is no host-written adapter to maintain — and the injected seam is exactly what pkc-js already holds, since `pkc.clients.libp2pJsClients[key]._helia` ([src/helia/libp2pjsClient.ts:20](https://github.com/pkcprotocol/pkc-js/blob/master/src/helia/libp2pjsClient.ts#L20)) *is* the Helia instance. It is private-by-convention and typed as raw Helia internals, though. Once this library's needs are firm, add a documented, version-stable accessor on pkc-js that returns that Helia node (carrying its gossipsub `libp2p.services.pubsub`, `blockstore`, and the registered-but-unexposed `fetch` service), rather than forcing consumers to reach through the churning `._helia` private field (for example the multiformats 13/14 split). pkc-js must register a gossipsub service on the node; without one, `PubsubVoter` construction throws `MissingPubsubError`. Filed as [pkc-js#183](https://github.com/pkcprotocol/pkc-js/issues/183), together with the `@libp2p/fetch` requirement and the tuning follow-ups below.

The forward-gate installs an **async** topic validator (a promise gossipsub awaits before `forwardMessage`), which needs no host-side gossipsub config — the pipeline simply does not resolve until it finishes. A fallback exists if that inline-await path is ever found to block gossipsub's message-processing loop unacceptably: switch to `asyncValidation: true` + `reportMessageValidationResult`, which requires the host to (a) construct gossipsub with `asyncValidation` enabled, (b) size the message cache (`mcacheLength` × `heartbeatInterval`) above the 10s validation budget so a slow validation is not evicted before it reports, and (c) tune `messageProcessingConcurrency`. Because those are gossipsub *constructor* options the host owns (this library only receives the running node), adopting the fallback is a documented pkc-js requirement, not a change this library can make alone. v1 uses the inline-await path and needs none of it. The **fetch service** (`@libp2p/fetch`) is a **required host service**: this library registers its own lookup function on it (answering `"bitsocial-votes/" + criteriaCid + "/root"` with the on-demand root record — see "Checkpoints") and runs its own requester against `libp2p.services.fetch` on the injected node, so the only pkc-js change is *adding the service to the shared node's construction* — `PubsubVoter` throws `MissingFetchError` at construction until it does, exactly as `MissingPubsubError` guards gossipsub.

**Gossipsub peer-score tuning against fresh-peer-ID RPC floods.** The forward-gate's `reject` lowers a sender's score, but libp2p peer IDs are free, so an attacker minting fresh IDs can still make honest peers spend one gate chain read per *novel* wallet (bounded, not eliminated — see "Transport", resource-exhaustion residual). Identity cannot be made costly; *network position* can. The levers live in the gossipsub **constructor**, which the host owns (this library only receives the running node): weight the invalid-message penalty (P₄), enable the IP-colocation penalty (P₆) so many peer IDs behind one address are scored together, and set publish thresholds / a newcomer ramp so an unproven peer gets only a small budget until it delivers valid messages. A further, ready-made lever is the **application-specific score (P₅)**: the forward-gate already classifies every peer's traffic as `accept` vs `reject`, which is exactly a per-peer reputation signal — this library can maintain that counter, but feeding it into scoring needs pkc-js to accept an `appSpecificScore` function at gossipsub construction (again host-owned). The hard **`maxTransmitSize`** message ceiling backing the inline-bundle live path (see "Message size cap") is one more constructor option in the same host-owned bucket. **Host gossipsub version floor:** the node must run **`@libp2p/gossipsub` >= 15.0.23**, the release that fixes CVE-2026-46679 (an unauthenticated subscription-flood memory DoS, ~22× amplification — a few MB of subscriptions crashes an unpatched node); this is a decode-limit fix in the gossipsub constructor path the host owns, not something the forward-gate can guard. In practice the host is well above the floor: pkc-js ships gossipsub `16.0.2`, and this library's integration test pins the latest, `16.0.3`. Track all of this as a pkc-js gossipsub-config item alongside the async-validation fallback above; none of it is a change this library can make alone.

**Optimistic (0-RTT) multistream-select for the cold-join fetch — host muxer/libp2p bound.** The cold-join root-record **fetch** costs ~0.9 s over a ~270 ms-RTT WAN link, and instrumenting it (`benchmark/cold-join.ts`, the `negotiate` vs `write→read` split) shows the **dominant term is the `connection.newStream` multistream-select handshake — ~2 RTT (~0.55–0.65 s)**, not the request/response (~1 RTT). libp2p already exposes the fix — `newStream({ negotiateFully: false })` optimistically writes the protocol + first bytes in one message, saving a round trip — but it is a **no-op in the host's pinned stack**: libp2p `3.3.4`'s `connection.newStream` passes an early single `protocol` to `muxer.createStream`, but yamux `8.0.1`'s `onCreateStream` ignores it (the muxed stream's `protocol` stays empty, so full `mss.select` always runs), and libp2p 3.3.4 never consumes `negotiateFully`. Because this library only *receives* the running node and never builds its muxer, the round trip cannot be removed here — it needs a host **muxer/libp2p upgrade** (a yamux that honours early-protocol negotiation, or a libp2p that threads `negotiateFully` through to `mss.select`). Same host-owned bucket as the gossipsub/fetch requirements above. The complementary shipped win that *was* in the library's control — collapsing the checkpoint **block** pull from two sequential round-trips to one via the piggybacked chunk index (see "The root record") — is already in.

**Provider-record announces to the HTTP routers — v1: the seeder's dedicated helia node announces its own records (this library, Node-only); kubo stays out of pubsub voting; querying needs no library config.** The two directions are asymmetric in the pinned stack. *Querying* rides the injected node: the host constructs it with one `delegatedRoutingV1HttpApiClientContentRouting({ url })` per configured router in `libp2p.services` (pkc-js `getDelegatedRoutingFields(httpRoutersOptions)`), and libp2p aggregates them into `libp2p.contentRouting`, so `findProviders` on ANY CID — the criteria CID at cold start today, chunk CIDs when the parallel chase-time lookup lands (see "Future improvements") — fans out to the host's routers with this library holding no router URLs at all. *Announcing* cannot ride that seam: the delegated-routing client's `contentRouting.provide()` is a no-op, so provider records only get written by a direct `PUT /routing/v1/providers` (kubo's JSON shape, `Providers[].Payload.{ID, Addrs, Keys}`; `Keys` is an array, so one PUT batches many CIDs). **v1 topology (decided 2026-07):** bitsocial-seeder runs a helia node **dedicated to pubsub voting** — constructed through a pkc-js instance's existing option seams (`libp2pOptions.services` / `heliaOptions`) or standalone; the library is agnostic, it takes the injected handle either way — carrying gossipsub, the fetch service (with the `maxInboundStreams` bump below), public listeners, **AutoTLS** (`@libp2p/auto-tls`, a `<peerId>.libp2p.direct` WSS certificate; needs AutoNAT-confirmed reachability and a persistent datastore + keychain for the cert) and a **WebRTC-Direct** listener (`@libp2p/webrtc`, certhash addrs, no CA) so browsers can dial it. **Kubo is deliberately uninvolved** — an earlier design (kubo recursively pins the checkpoint root, its Reprovider + the addresses-rewriter proxy keep block records fresh) was considered and rejected: it needs a pin-update/GC lifecycle, its per-CID records lag the live root after chunk reflow, the criteria-CID record still cannot come from kubo (wrong identity; kubo neither joins the topic nor answers the fetch-protocol root pull), and the two-node glue outweighs one HTTP PUT from the node that actually holds the state. **Announcing is therefore this library's job** (implemented 2026-07, `src/transport/announce/`): a Node-only announcer gated on `PubsubVoterOptions.httpRouterUrls?: string[]` (absent/empty ⇒ never announce; the browser build swaps in an inert stub via the same package.json `browser` remap as `src/storage/`), PUTting one batched **unsigned** record per router — `ID` = the injected node's peer id, `Addrs` = `libp2p.getMultiaddrs()` filtered client-side to public IP + DNS addrs (dialable precisely because AutoTLS/WebRTC-Direct publish their addrs into the address manager — no rewriter needed; an announce with no surviving address is skipped, matching the production router, which drops addr-less providers), `Keys` = every joined contest's criteria CID + current root + chunk CIDs, deduplicated. Cadence: hourly (pkc-js's `providePubsubTopicRoutingCidsIfNeeded` cadence — a 24× margin under the production router's 24 h record TTL, and a seam to revisit if that TTL shrinks toward the tracker-style ~30 min its code TODO aspires to) plus a coalesced re-announce — debounced 10 s, so a gossip burst costs one announce per window — on checkpoint changes (riding the same transition that dirties the on-demand encode cache), on topic joins, and on `self:peer:update` (the address set changed: an AutoTLS cert landing, a WebRTC-Direct certhash rotation). Lifecycle mirrors the lazy fetch responder: the first joined topic starts it, the last left topic stops it (existing records age out by TTL; there is no un-announce). Best-effort per router: a failing or slow router (10 s PUT deadline) is isolated and never retried — the next tick covers it. Unsigned is correct against the production router (pkc-http-router reads only `Payload.{ID, Addrs, Keys, AdvisoryTTL}`; no signature field exists), and its source-IP anti-spoofing (`cleanAddrs`) is passed naturally by a seeder announcing its own addresses — with the dual-stack caveat that a PUT arriving over IPv4 silently drops `/ip6` addrs (and vice versa), while DNS addrs pass through unvalidated. The cold-join WAN bench exercises the path end-to-end: the orchestrator hosts the mock router locally, reverse-tunnels it to the seeder host (`ssh -R` — out-of-band of the measured join path, so tunnel latency distorts nothing), the seeder announces for real, and the joiner discovers it from the announced record; no hardcoded provider record remains. The standing deployment requirement: **the seeder's helia node must be publicly dialable**, which the browser-first pkc-js default does not do on its own — the announcer enforces the spirit of this by refusing to announce private/loopback-only address sets.

**Optional: raise the fetch service's `maxInboundStreams` on the shared node — a directory-scale speedup, no longer required for correctness.** A real client (e.g. 5chan.app) cold-loads a whole **directory** of contests at once: it joins all ~63 boards concurrently, and each board's cold-start fires one **fetch** for its root record (see "Checkpoints → pull"). Those fetches land as concurrent *inbound* streams on the **shared** seeder — and libp2p `3.3.4` registers every protocol handler with `DEFAULT_MAX_INBOUND_STREAMS = 32` (`node_modules/libp2p/dist/src/registrar.js`). `@libp2p/fetch` passes no override, so the node serves the first 32 and **resets** the rest (`TooManyInboundProtocolStreamsError` → `muxedStream.abort`, `connection.js`). Measured on the WAN rig (`benchmark/directory-{seeder,join}.ts`, `npm run bench:directory-load`): a naive all-at-once join of 63 boards against a **default-capped** node converges **63/63 in ~6.3 s** — because the cold-start now **retries** a reset fetch until a deadline (`#fetchRootWithRetry`, see "Checkpoints → pull"), riding out the saturated cap until every board wins a slot. The node-side limit is therefore no longer required for the directory to load. It is still a worthwhile *speedup*: registering the fetch service with a generous limit — `fetchService({ maxInboundStreams: 1024, maxOutboundStreams: 1024 })`, sized above the expected concurrent-board fan-in of one client (the caps are per connection, so concurrent *users* do not share slots) — lets the same join finish in ~8.6 s with no resets at all (reproduce with `BENCH_FETCH_MAX_STREAMS`). Helia's bitswap already defaults to `maxInboundStreams = 1024`, so the checkpoint *chase* is not a wall; only the fetch handler's default of 32 matters, and the client now handles it either way.

## Future improvements

Combining several rules into one criteria slot is decided in principle but deferred: v1 (Pass gate + `constant` weight) and the weighted example (Pass gate + single-asset weight) each use exactly one rule per slot, so nothing ships a combinator yet. The direction is *structural* — mirroring pkc-js `settings.challenges`, a list the runtime iterates — not a general recursive combinator rule.

- **Gate combining — a `challenges`-style AND-array.** Make the `rule` slot `RuleRef[]`, every term required (each scores `> 0`), iterated by the runtime exactly as pkc-js ANDs its `challenges`. This expresses "hold a Pass **and** ≥ N BSO" with no combinator and no recursion. Building it: `CriteriaSchema.rule` becomes a non-empty array (normalized before canonical encoding, so one logical rule stays one topic), `validateCriteriaRules` iterates it, and `ChainReadContext` evolves from a single pre-resolved `chain` to a `chainFor(ticker)` resolver so each gate can name its own chain. An **OR**/nested gate, if ever needed, is a small *gate-only* `any` grouping decided then — still not a general combinator. (pkc-js also ORs via `exclude` indices, but every `exclude` dimension — `role`, `postScore`, `rateLimit`, account age — is a social/author signal absent for a bare voting wallet, so only the flat AND-array transfers.)
- **Weight combining — a weight-only, non-recursive reduction.** For multi-asset additive weight (e.g. `#Passes + BSO/1000`) add a reduction over weight-slot terms (a `sum`; `min`/`max` only if a real criteria needs them). It is weight-only — the gate has the canonical AND reduction, weight has none, which is exactly why weight must *name* its operator — and non-recursive (a flat list of leaves, not a tree of combinators). It also requires resolving the lazy-tally upper bound for balance-derived weight (see "Open questions"), since a summed balance carries no free wire-side ceiling.

Neither introduces a single "combinator" abstraction into the core: the gate composes by AND-listing, weight by a named reduction, and no rule recurses into the registry.

- **Directed-session block pull with a parallel router lookup.** Two chase-side upgrades investigated together (2026-07): (1) pull each chased root's blocks through `helia.blockstore.createSession(root, { providers: [advertiser] })` instead of per-block `blockstore.get` — the plain path fires an unconditional `findAndConnect` routing query per block (pure waste while nothing announces chunk CIDs) and broadcasts WANT-BLOCK to every connected peer; a session seeded with the peer(s) that advertised the root sends targeted wants only (the pkc-js PR #191 pattern — the advertiser is always known: cold-start `pull(peer)` has it in hand, and the gate's `onRootRecord(record, from)` already carries `from`, currently dropped at the voter's wiring). The per-`get` `providers` option in `@helia/interface` is a dead end — `Bitswap.want()` ignores it in the pinned helia. Feature-detect `createSession` (unit-test blockstores lack it) and fall back to the broadcast `get` on session failure. (2) With seeders announcing root/chunk provider records (see "Deferred pkc-js work"), keep the session's `maxProviders` above the seed count so its background `findNewProviders` queries the HTTP routers **in parallel** with the seeded wants — a provider found there (another seeder; converged seeders share identical chunk CIDs, and a block provider need not be a topic subscriber) serves chunks over plain bitswap, so a joiner survives its advertiser dropping mid-chase. Re-run the WAN bench on both changes.
- **Chainless rules — the `chains` seam becomes optional, but the crash moves, it does not vanish.** A voting contest need not touch a blockchain (e.g. a `constant`-only rule), so the `chains: ChainClientFactory` seam should become optional, exactly as `signer?` already is. **Ownership decision:** whether a contest reads chain is a property of its *rule*, declared canonically by `criteria.requires.chains` (an empty record = chainless — it stays required-but-possibly-empty so intent is read from the topic-affecting criteria, not inferred from a runtime injection). The guard stays a *single* choke point, not a per-rule `if (!chain) throw`: every read already funnels through `#chainFor(ticker)` ([src/client/voter.ts, `#chainFor`](./src/client/voter.ts)), which throws when no client is configured. When `chains` is undefined, route that seam through a stub factory that throws one clear "no chains factory was injected" — chainless rules never call it and work; chain rules get one well-worded error in one place. But making the seam optional *alone* only relocates the failure: the constructor eagerly assumes a **gating chain** — it resolves `ruleTicker → ruleChain` and throws if absent, bakes `#chainId = requires.chains[ruleTicker].chainId` into **every EIP-712 ballot signature**, uses the gating chain as the tie-break block-hash seed, mandates `chain: ChainClient` in `ChainReadContext` (`src/rules/types.ts`), and `tickerForRef` throws on an empty `requires.chains` (`src/chain/ticker.ts`). So a genuinely chainless contest first needs three consensus-critical semantics decided: what `chainId` means in the EIP-712 `domain` with no chain, what seeds the tie-break, and what block (if any) a chainless rule is sampled at. **Sequencing:** do not loosen the type ahead of those answers — v1 is NFT-only (`erc721-min-balance` gate + `constant` weight, both chain-configured), so a `chains?` type would advertise a chainless path the constructor cannot honour, and a type that lies is worse than a required seam. Make `chains?` optional *and* make the gating-chain / `chainId` / tie-break path conditional-on-rule in the same change that ships the first chainless rule. This shares the "single pre-resolved chain → `chainFor` resolver" evolution the gate-combining bullet above already needs.

## Open questions

- **Lazy-tally upper bounds for non-constant weight.** Moved to [ROADMAP.md, Deferred](./ROADMAP.md#deferred-designed-not-shipped) — it is a precondition for shipping balance-derived weight, which is out of v1, so it lives with that deferred work rather than as a standing open question here.
- **Pinned-block name resolution — API gap closed upstream, multi-chain block-selection still open.** Community-name verification (see "Tally") should read the naming registry at a canonical historical block, exactly as balance reads are pinned to the bucket block. The resolver-API half is now solved upstream: [bso-resolver#3](https://github.com/bitsocialnet/bso-resolver/issues/3) is resolved (PR #4), so `@bitsocial/bso-resolver`'s `resolve` takes an optional `blockNumber?: bigint` (omitted ⇒ head) and the injected `NameResolver` seam here advertises it. But pinning was never *only* an API gap: the registry lives on its own chain (Ethereum for `.bso`) while the bucket boundary block is defined on the criteria's chain (Base in the 5chan example), so a deterministic read still needs a canonical per-bucket block *on the registry's chain* (per-chain bucket math, or mapping the bucket boundary timestamp to the nearest registry-chain block — the same multi-chain problem as the tie seed and the gate AND-array). That half remains open, so **v1 still resolves at head** (leaves `blockNumber` unset) and accepts a disagreement window around a re-point: two verifiers can transiently disagree about a name whose record just changed — the same eventual consistency the CRDT accepts elsewhere, bounded in practice because records change rarely and a refreshing client re-signs on the recommended liveness cadence (`ceil(voteExpiryBuckets / 2)` buckets), so for such clients the window is at most one republish cycle, up to half the expiry window, not a single bucket (a client that never refreshes just lets the vote expire). Passing the pinned block is a one-line change once the registry-chain bucket block is chosen.
- **Checkpoint byte-vector — resolved.** The byte layout (a canonical dag-cbor array of the current winners, sorted by address, inlined, size-cap chunked under one root CID) is pinned by fixed root-CID vectors in `src/checkpoint/codec.test.ts` (a known winner set, a multi-chunk split under a small cap, and the empty set), so independent seeders agree on the root CID. Only the **HAMT-keyed variant** for light-client single-wallet lookups / absence proofs remains deferred (see "A sorted list, not a key-map").
